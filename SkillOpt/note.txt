## 一、 前言 (Introduction)

### 1.1 研究背景與實務痛點
在建構 LLM Agent 時，我們經常需要針對特定領域進行適應（Domain Adaptation）。然而，直接微調（Fine-tuning）模型權重往往因封閉型 API 限制或高昂算力成本而不可行。實務上，工程師多採用自然語言寫成的「技能文件（Skill Document）」（如系統提示詞或操作指南）作為 Agent 的外部適應層。

然而，現有的技能優化方法存在嚴重的技術債：
* **手寫或單次生成**：耗時且難以涵蓋複雜的邊角案例。
* **無約束的自我反思（如 Reflexion、EvoSkill 等）**：缺乏「步長（Step size）」與「驗證（Validation）」控制，優化器 LLM 容易因過度擬合單一失敗而大幅改寫提示詞，造成**文字空間的災難性遺忘**與 **Prompt 膨脹**。

### 1.2 SkillOpt 的解決思路與本筆記目的
微軟提出的 **SkillOpt** 框架，其核心思想是將「文字技能修改」視為一個嚴格、受控的機器學習優化過程。它將技能文件（`best_skill.md`）視為外部可變參數，並將深度學習中的經典機制（批次處理、學習率、梯度裁剪、驗證集、動量）映射至文字編輯上，實現離線、零部署成本的穩定優化。

本筆記旨在為 AI 軟體工程師客觀拆解 SkillOpt 的系統架構、底層 JSON 資料結構、提示詞合約（Prompt Contracts）以及演算法執行線路，作為未來開發自適應 Prompt 架構的工程參考指南。

## 二、 系統基礎架構與問題定義 (Phase 1: Foundations & Formulation)

本章節釐清 SkillOpt 的底層架構設計、系統變數與評估機制。SkillOpt 的核心原則是將「技能文件（Skill Document）」定義為外部可讀寫的參數狀態，並透過明確的分工與資料隔離來確保優化軌跡的穩定性。

### 2.1 雙模型架構：職責分離的學生與教練

在傳統的參數微調中，模型自身同時扮演「執行者」與「被修改者」的角色。SkillOpt 將這兩個職責完全解耦，拆分為兩個獨立的模型實體：

1. **目標模型 (Target Model, 符號 $M$)**：
   * **角色定義**：執行任務的「學生」。
   * **狀態特徵**：在整個優化過程中，其權重（Weights）與原生系統提示詞（System Prompts）皆**保持凍結（Frozen）**。
   * **運行環境**：執行於特定的任務框架（Execution Harness, $h$）中。
   * **數學定義**：給定一個任務 $x$ 與當前技能 $s$，目標模型在框架中運行並輸出執行軌跡 $\tau(s)$ 與標量評分 $r(s) \in [0, 1]$：
     $$(\tau(s), r(s)) = h(M, x, s)$$
2. **優化器模型 (Optimizer Model, 符號 $O$)**：
   * **角色定義**：修改外部狀態的「教練」。
   * **狀態特徵**：通常採用能力較強的 Frontier Model（如 GPT-5.5 [35]），僅在離線訓練階段（Offline Training Phase）運行。
   * **運行環境**：不直接參與任務執行，其職責為讀取 $\tau(s)$ 與 $r(s)$，並輸出針對技能文件的結構化修改建議（Edits）。

**💡 軟體工程實務價值**：這種職責分離的設計確保了**零部署成本（Zero Inference Cost）**。在開發階段，雖然優化器模型會消耗大量的 API Token，但一旦離線訓練結束並導出最終的技能文件，在正式環境部署時，僅需單獨運行凍結的目標模型加文字檔案。

---

### 2.2 作為外部 Trainable State 的技能文件

在 SkillOpt 中，被訓練與優化的參數實體是一份名為 **`best_skill.md`** 的純文字 Markdown 檔案（長度約 300 至 2,000 tokens）。

* **注入機制**：在問答任務中，該文件會直接串接（Prepend）至系統或開發者指令的前面；在工具操作任務（如 Codex [38] 或 Claude Code [39]）中，它會被寫入工作區，作為 Agent 的持久化程序記憶（Persistent Procedural Memory）。
* **承載內容**：包含通用操作步驟、領域啟發式規則（Heuristics）、工具調用規範、輸出格式約束以及常見失敗模式的防禦性指南。

---

### 2.3 機器學習式的資料隔離與盲測合約

為防止 LLM 發揮其強大的「作弊（過度擬合）」能力，SkillOpt 建立了嚴格的資料隔離合約：

1. **訓練集 ($D_{tr}$ / Train)**：
   * **合約內容**：提供優化器模型可見的軌跡（Rollout evidence）。目標模型在此執行任務，產生的錯誤與成功案例將作為修改的「依據」。
2. **選擇/驗證集 ($D_{sel}$ / Selection)**：
   * **合約內容**：優化器模型**絕對不可見**其中的任務與軌跡。
   * **驗證守門員（Validation Gate）機制**：任何修改後的候選技能 $\tilde{s}$，必須在 $D_{sel}$ 上進行獨立測試，取得驗證分數：
     $$s_{sel}^* = \arg\max_{s \in \mathcal{C}(D_{tr})} \frac{1}{|D_{sel}|} \sum_{x \in D_{sel}} r(s)$$
   * **嚴格大於規則**：唯有當驗證分數**嚴格大於（Strictly Greater than）**當前基準分數時（平手亦會被拒絕），修改才會被寫入 `best_skill.md`。這有效防堵了「看似合理但有害」的提示詞修改積少成多。
3. **測試集 ($D_{test}$ / Test)**：
   * **合約內容**：完全鎖定，直到最終報告階段。用於檢驗最終技能的泛化能力。

## 三、 核心優化循環（上）：資料流收集與平行診斷 (Phase 2: Data Flow & Parallel Reflection)

本章節介紹 SkillOpt 如何實作「資料流的前向收集與反向診斷」。為了解決以往提示詞優化因「單一軌跡反思」產生的特例修復問題，本框架建立了基於 Batch 的收集機制，並在反向診斷中採用 MapReduce 式的平行小批次處理。

---

### 3.1 前向傳播 (Forward Pass)：Rollout Batch 的統計顯著性

在優化的第一步，目標模型需要帶著當前技能 $s_{cur}$ 收集執行證據。

* **大批次設計**：SkillOpt 預設在每一步（Step）使用 **Rollout Batch Size $B = 40$**。這意味著目標模型會同時在訓練集 $D_{tr}$ 中跑 40 個獨立的任務。
* **避開「特例修復」**：若每次只看單一失敗軌跡，LLM 通常會對當下的「隨機誤差」（例如網絡延遲、特定極端輸入）進行過度擬合（Overfitting）。使用 $B = 40$ 的大批次能提供統計上的顯著性，確保後續診斷出的問題是「反覆出現的系統性錯誤（Recurring patterns）」。

---

### 3.2 資料流的分流機制：成功池與失敗池

當 40 筆執行軌跡收集完成後，系統會自動根據得分 $r(s) \in [0, 1]$ 進行精確分類：
* **失敗池 (Failure Pool)**：分數未達標（例如 $r < 1.0$）的軌跡。
* **成功池 (Success Pool)**：分數達標（例如 $r = 1.0$）的軌跡。

**💡 軟體工程考量**：成功與失敗的優化語境完全不同。失敗池旨在尋找**漏洞並進行糾正**（Corrective edits）；成功池旨在尋找**未被明文記錄的好習慣並加以鞏固**（Reinforcement edits）。分流機制避免了不同優化目標在同一個 Prompt 中產生邏輯混淆。

---

### 3.3 反向診斷 (Backward Pass)：並行 Minibatch 反思與 MapReduce 類比

為了處理分流後的軌跡池，系統引入了 **Minibatch Size $B_m = 8$** 的切分機制。

#### 1. MapReduce 視角下的優化設計
如果將 20 個失敗軌跡一次全部塞進 LLM，會因為上下文過長（Context Bloat）導致模型漏掉關鍵細節（Lost in the middle）。因此，SkillOpt 採用了類似 MapReduce 的平行運作架構：

```
[40 Traces] ──> 分流 ──> [20 Failures] ──> 切片 (Bm=8) ──> [Minibatch 1 (8)] ──> 🧠 呼叫 Analyst 1 (Map)
                                                       ──> [Minibatch 2 (8)] ──> 🧠 呼叫 Analyst 2 (Map)
                                                       ──> [Minibatch 3 (4)] ──> 🧠 呼叫 Analyst 3 (Map)
```

* **Map 階段（並行分析）**：系統會啟動多達 16 個平行的 Analyst Workers（API 呼叫）。每個 Worker 每次只會分到一個大小最多為 8 的小批次。
* **為什麼 $B_m = 8$ 是最佳折衷？**
  * $B_m = 1$：過度擬合單一軌跡。
  * $B_m = 8$：在 LLM 的 Context 限制與推理負載之間取得平衡，強制模型「橫向對比」這 8 個軌跡，提取出「多數任務共同跌倒的地方」。

---

### 3.4 提示詞合約 (Prompt Contracts) 與 JSON 資料規格

為了讓並行分析能被程式自動化解析與執行，並行分析師的輸出被規範為嚴格的 JSON 格式。以下為論文附錄中兩個最核心的分析 Prompt 設計：

#### 1. 失敗診斷合約 (System Prompt: `analyst_error.md` [C.2.1])
* **輸入**：8 個失敗任務的執行軌跡、當前的 `best_skill.md`。
* **核心規則**：找出這 8 筆資料中最嚴重的**共同失敗模式（COMMON failure patterns）**，不要修改保護區（Delimited by `<!-- SLOW_UPDATE -->`）。 Edits 必須具備通用性。
* **JSON 輸出結構**：
```json
{
  "batch_size": 8,
  "failure_summary": [
    {
      "failure_type": "string",  // 錯誤分類，如 '工具調用異常'
      "count": 5,                // 發生次數
      "description": "string"    // 具體錯誤描述
    }
  ],
  "patch": {
    "reasoning": "string",       // Chain-of-Thought 修改動機
    "edits": [
      {
        "op": "append | insert_after | replace | delete", // 支援的原子操作
        "target": "string",      // 精確匹配的定位原文字串 (append 時為空)
        "content": "string"      // 擬寫入的新文字
      }
    ]
  }
}
```

#### 2. 成功強化合約 (System Prompt: `analyst_success.md` [C.2.2])
* **輸入**：8 個成功任務的執行軌跡、當前的 `best_skill.md`。
* **核心規則**：找出這 8 筆資料中**已經起作用、但尚未寫入 `best_skill.md`** 的通用好習慣。偏好「強化現有段落」，保持保守與精簡。
* **JSON 輸出結構**：
```json
{
  "batch_size": 8,
  "success_patterns": ["string"], // 觀察到的高效行為模式
  "patch": {
    "reasoning": "string",
    "edits": [
      {
        "op": "append | insert_after | replace | delete",
        "target": "string",
        "content": "string"
      }
    ]
  }
}
```

## 四、 核心優化循環（中）：階層式合併與衝突解決 (Phase 2 Continued: Hierarchical Merge)

當並行分析師（Analyst Workers）各自處理完一個小批次（Minibatch）後，系統會回收多份包含結構化 Edits 的 JSON 提案。為了解決提案之間的冗餘、矛盾以及字串定位衝突，SkillOpt 實作了雙階段的「階層式合併」機制。

---

### 4.1 樹狀合併機制 (Tree Reduce) 與 Merge Batch Size

在實務中，若單次 Step 產生的 Minibatches 數量過多（例如有 10 份以上的 JSON 提案），直接將所有提案塞入單一 Prompt 中會導致 LLM 的注意力分散與邏輯混亂。

#### 1. Merge Batch Size = 8 的工程約束
為了解決這個問題，SkillOpt 設定了 **Merge Batch Size = 8** 的合併上限（一次合併最多隻處理 8 個 JSON 檔案）。當提案數量 $N > 8$ 時，系統會自動採取樹狀結構（Tree Reduce）進行多輪合併：

```
[10 個原始 JSON 提案]
   │
   ├─ 隨機取出前 8 個 ─> 🧠 呼叫 merge_failure.md (第一輪) ─> [暫存 Merged_A]
   │                                                             │
   └─ 剩下 2 個 ────────────────── 同時併入第二輪 ───────────────┼─> 🧠 呼叫 merge_failure.md (第二輪) ─> [最終 Failure JSON]
```

* **Round 1**：處理前 8 份 JSON 提案，經由 `merge_failure.md` 合併為 1 份暫存提案 `Merged_A`。
* **Round 2**：將 `Merged_A` 與剩下的 2 份提案（共 3 份）再次送入 `merge_failure.md` 合併，產出最終的失敗修改清單。

**💡 系統可擴展性（Scalability）**：此設計使優化管線（Pipeline）不受資料量限制。即使在高併發環境下產生數十份提案，系統皆能透過此樹狀合併合約，穩定且無失真地收斂至單一輸出。

---

### 4.2 第一階段：同溫層內部合併 (Pre-merge)

此階段將「失敗組」與「成功組」的提案分別在各自的語境中進行初步整合。

#### 1. 失敗草案合併 (`merge_failure.md` [C.2.3])
* **輸入**：多份（最多 8 份）獨立的失敗修改 JSON。
* **合併原則**：
  * **去重與支持度累加 (`support_count`)**：若多個分析師針對同一錯誤提出類似修改，保留措辭最通用的一版，並將其 `support_count` 設為原始提案的累加值。
  * **解決衝突 (Conflict Resolution)**：若針對同一點的修改產生矛盾，依據分析師的理由強度進行裁決或合成。
  * **獨立性檢查 (Independence)**：確保最終清單中，沒有兩個 Edit 操作指向 `best_skill.md` 的同一個字串定位點，避免在打 Patch 時發生覆蓋衝突。

#### 2. 成功草案合併 (`merge_success.md` [C.2.4])
* **輸入**：多份（最多 8 份）獨立的成功修改 JSON。
* **合併原則**：
  * **保守過濾**：比對當前的 `best_skill.md`，若提案中的優良行為已經在現有技能中被描述，則直接將該 Edit 剔除。
  * 統計累加 `support_count`，並將 `"source_type"` 標記為 `"success"`。

---

### 4.3 第二階段：跨堆最終決選合併 (Final Merge)

在產出一份精煉的「失敗合併清單」與一份「成功合併清單」後，系統會調用最終的協調器進行「跨堆合併」。

#### 1. 最終決選規則 (`merge_final.md` [C.2.5])
* **輸入**：第一階段產出的 Unified Failure Patch 與 Unified Success Patch。
* **合併規則**：
  * **最高原則：失敗修復優先 (FAILURE PATCHES TAKE PRIORITY)**。解決漏洞（Bug-fixing）的權重大於錦上添花。
  * **定位衝突解決**：若失敗 Edit 與成功 Edit 試圖修改同一個 `target`（定位點），**無條件保留失敗 Edit 版本的內容與修改操作**。
  * **保留不衝突的成功洞見**：將與失敗清單不產生定位衝突、不互相矛盾的成功 Edit 合併進來，作為技能的防禦性強化。
  * **傳遞元數據 (Carry Forward Metadata)**：最終輸出的每一個 Edit 成員，都必須保留其 `support_count` 與 `source_type`（"failure" 或 "success"），供下一個階段（打分與排序）做為篩選依據。

## 五、 核心優化循環（下）：有界更新、打分排序與守門員驗證 (Phase 3: Bounded Updates & Validation)

本章節介紹 SkillOpt 如何實作「文字空間的梯度裁剪與嚴格驗證」。為了防止優化過程因 LLM 的隨意重寫而崩潰，本框架引入了編輯預算限制、精確的打分排序，以及基於獨立驗證集的盲測淘汰機制。

---

### 5.1 文字版學習率：編輯預算與 Cosine Schedule 衰減

在傳統機器學習中，學習率（Learning Rate）控制了參數更新的步長。若步長過大，模型將無法收斂。在文字空間中，SkillOpt 使用 **編輯預算 (Edit Budget, 符號 $L_t$)** 來作為文字版的學習率。

* **有界更新 (Bounded Updates) 的重要性**：
  系統限制單次步驟（Step $t$）允許套用的最大 Edit 數量為 $L_t$。即使合併後的清單中有多個高價值的修改建議，系統也絕不超額套用。這保證了技能文件的演進具有**連續性**，防止單次修改幅度過大引發災難性遺忘。
* **餘弦衰減排程 (Cosine Decay Schedule)**：
  預算 $L_t$ 會隨著訓練進程進行衰減（常規設定從 $L_t = 4$ 逐漸衰減至 $L_t = 2$）：
  * **優化初期**：$L_t$ 較大，允許對技能文件進行大刀闊斧的結構性修改（Exploration）。
  * **優化後期**：$L_t$ 較小，限制系統僅能進行用詞微調與邏輯修補（Consolidation）。

---

### 5.2 候選修改之打分與排序合約

當我們手上有 $N$ 個經由合併產出的 Edit 提案，但本回合預算僅有 $L_t$ 時，系統會調用專用的排序器進行篩選。

#### 1. 排序器評估準則 (`ranking.md` [C.2.6])
排序器會根據以下四個絕對優先級標準（由高至低）進行打分，並輸出保留的 Edit 索引值：
1. **系統性影響力 (Systematic impact)**：能修復高頻、反覆出現之錯誤的 Edit 優先。此準則直接掛鉤前一階段累加的 `support_count`。
2. **互補性 (Complementarity)**：優先選擇填補現有 Skill 漏洞的修改，若與現有規則語意重複則降低優先級。
3. **通用性 (Generality)**：傾向於保留表述為「通用程序指南」的修改，排除過度擬合特定任務（如寫死特定檔名或數值）的修改。
4. **可操作性 (Actionability)**：指引具體、步驟清晰（有明確工具或條件分支）的 Edit 優先於空泛的建議。

* **排序器輸出 JSON 結構**：
```json
{
  "reasoning": "string",
  "selected_indices": [0, 2] // 根據預算 L_t，僅輸出前 L_t 個高優先級的 Edit 索引
}
```

---

### 5.3 驗證集守門員與嚴格平手淘汰制

套用 Top $L_t$ 個修改後的候選技能 $\tilde{s}$，必須在獨立的驗證集（Selection Split, $D_{sel}$）上進行跑分測試。

* **盲測防禦**：目標模型在 $D_{sel}$ 上的評估是完全自動且與優化器隔離的。這能有效篩選出那些「聽起來非常有道理，但目標模型讀不懂或執行變差」的有害修改。
* **平手淘汰制 (Strictly Greater)**：
  驗證合約規定，只有當新技能的分數**嚴格大於**舊技能時（$score_{cand} > score_{cur}$），修改才會被寫入硬碟並更新。若分數不變（平手）或下降，則一律判定為「拒絕套用（Reject）」。這防止了無意義規則在技能文件中無限累積（Prompt Bloat）。

---

### 5.4 被拒修改緩衝區 (Rejected-Edit Buffer) 的狀態生命週期與負面回饋

當一個修改被驗證集拒絕時，系統不會直接丟棄，而是將其記錄在一個 **Epoch-local 暫存區（Step Buffer, $\mathcal{B}$）** 中，作為後續優化的「錯題本」。

* **數據結構與內容**：
  Buffer 內以結構化的方式儲存：**觀察到的失敗模式**、**這次試圖新增/替換的具體字串**，以及**在驗證集上造成的分數下降量**（如 `-0.15` 或 `0.00`）。
* **負面回饋注入**：
  在同一個 Epoch 內，當進入下一步（Step $t+1$）的 `analyst_error.md` 與 `ranking.md` 分析時，系統會將這個 Buffer 的失敗歷史轉為文字，注入到 Prompt 的最開頭，警告模型：「*我們最近嘗試過以下修改，但導致了驗證分數下降/平手，請避開這些失敗思路。*」這在工程上有效避免了 LLM 陷入「重複嘗試相同錯誤修改」的無效循環中。

---

### 5.5 【圖表插入點】：系統完整管線與容錯機制

> **【工程師提示：請在此處插入論文的 Figure 2（Pipeline of SkillOpt）】**
> 本圖展示了 Rollout Batch -> Minibatch 反思 -> 階層式合併 -> 排名剪裁 -> 驗證守門員與被拒緩衝區的完整優化 Pipeline。

在將 Edit 候選人套用到實體 Markdown 檔案時，LLM 可能會產生微小的字串對齊偏差。為此，SkillOpt 實作了工程容錯：

* **精確匹配與跳過 (Skip)**：系統堅持精確匹配（Exact Match）定位字串 `target`。若因 LLM 幻覺導致匹配失敗，系統不崩潰，而是將該單步 Edit 標記為 `skip`，其餘無衝突的 Edit 照常執行。
* **日誌追蹤 (`edit_apply_report.json`)**：每一次套用修改，系統皆會導出該 JSON 文件，詳細記錄每一個 Edit 最終是被 `applied`（成功套用）還是 `skipped`（字串匹配失敗），確保整份 `best_skill.md` 的演進歷史完全可追溯、可審計。
* **驗證回滾 (Rollback)**：若因為部分 Edit 被跳過，導致剩餘套用的規則邏輯殘缺，驗證集（Selection Split）會測出分數下降，Validation Gate 就會自動觸發系統 Rollback（回滾），將技能文件恢復至修改前的備份版本。

## 六、 長期記憶與教練進化：Epoch 邊界的宏觀調控 (Phase 4: Epoch-level & Meta Updates)

在微觀的「單步更新」中，系統容易流於見招拆招，而遺忘大局。為了平衡短期的靈活修補與長期的穩定發展，SkillOpt 在每個 Epoch（訓練週期）結束時，會暫停小步微調，啟動宏觀的長線調控。

這部分由兩個機組成：**慢速更新（寫給學生看）** 與 **元技能（教練自己看）**。

---

### 6.1 縱向對比 (Longitudinal Comparison)：同一份 20 題的考卷

要找出長期的趨勢，必須進行科學的對照實驗。

* **「控制變因」的對比法**：
  在 Epoch 結束時，系統會從訓練集中隨機抽出 **20 個任務**（就像從題庫隨機抽 20 題組成一份考卷）。
  系統會讓目標模型分別帶著「上一個 Epoch 的舊技能」與「這一個 Epoch 的最新技能」，去考這同一份 20 題的考卷。
* **分類成績單**：
  比對這 20 題的對錯後，系統會自動將題目分類為四種狀態：
  1. **進步 (Improvements)**：本來做錯，新技能做對了。
  2. **退步 (Regressions)**：本來做對，新技能卻做錯了（這在優化中是最危險的警訊）。
  3. **持續失敗 (Persistent failures)**：新舊技能都做錯，代表死穴還沒被解開。
  4. **穩定成功 (Stable successes)**：新舊技能都做對。

---

### 6.2 慢速更新 (Slow Update)：給學生的長期戰略保護區

戰略顧問（`slow_update.md` [C.2.7]）會閱讀這份 20 題的縱向成績單，分析為什麼會產生「退步」，並撰寫巨觀的指導原則。

* **唯讀保護區 (Protected Section)**：
  這段指導原則會被寫入 `best_skill.md` 檔案中，並用 Markdown 註解 `<!-- SLOW_UPDATE_START -->` 與 `<!-- SLOW_UPDATE_END -->` 包裹起來。
  在接下來的 Epoch 中，單步優化器（也就是 Phase 2 那些小分析師）對這個區塊**只有唯讀（READ-ONLY）權限**，絕對不准修改。這確保了宏觀的長期教訓不會被短期的微調給擦除。
* **驗證安全閥 (Validation Check)**：
  戰略顧問寫的原則，可能也會幫倒忙。因此，加入了保護區文字的技能，**必須再去跑一次驗證集（$D_{sel}$）**。唯有分數上升才會被正式寫入；若分數變低，這次的慢速更新就會被無情拋棄，技能文件直接回復（Rollback）到加上保護區之前的狀態。

---

### 6.3 教練專屬秘笈 (Meta Skill)：不對外公開的內功心法

這是 SkillOpt 在工程架構上最精妙的一手。系統會請「優化教練（`meta_skill.md` [C.2.8]）」來寫一本**「給未來教練看的工作手冊」**。

* **不對外公開的記憶**：
  這份元技能（Meta Skill）**絕對不會寫入 `best_skill.md`**，目標模型（學生）一輩子都看不到它，這保證了部署時的 Prompt 維持極度精簡。
* **如何影響未來**：
  在下一個 Epoch 中，當單步優化器（分析師、合併員、排序器）被呼叫時，這份元技能會被**直接貼在這些 Optimizer API 的 System Prompt 最前面**。
  教練看過這份工作手冊後，例如手冊上寫著：*「在這個任務中，寫死欄位名稱通常會被驗證集拒絕，請多寫通用的程序指南。」*，未來的教練就會自動避開地雷，提出更高品質的 Edit。

---

### 6.4 類比深度學習優化組件

為方便工程師理解，我們可以將這兩個機制與深度學習中的宏觀穩定機制進行對比：

| 深度學習組件 | SkillOpt 對應機制 | 簡單白話解釋 |
| :--- | :--- | :--- |
| **動量 (Momentum) / 指數移動平均 (EMA)** | **慢速更新 (Slow Update 保護區)** | 捕捉長期軌跡的大方向，防止單步的梯度（文字修改）引發震盪，造成前功盡棄（退步）。 |
| **元學習 (Meta-Learning / Learning to learn)** | **元技能 (Meta Skill 教練秘笈)** | 優化器自己也在學習「在這個環境下，該如何提出更容易通過驗證的修改建議」。 |

## 七、 實務算法執行全景流程 (Algorithm Full Flowchart)

本章節以高密度的系統工程思維，巨細靡遺地拆解 SkillOpt 演算法的每一行程式碼執行線路，並明確標註 **LLM 的調用時機、模型角色** 以及 **對應的 System Prompt 文件名稱**。

---

### 7.1 演算法全景文字流程圖 (Pipeline Flow)

```
[初始化] ──> 計算初始 s0 分數 (Target Model)
  │
  └───> 【Epoch 循環開始】 (e = 1...E)
          │  重置 Step 緩衝區 B
          │
          └───> 【Step 循環開始】 (在當前 Epoch 的訓練資料內跑 batch)
                  │
                  ├──> [1. 前向 Rollout] 🤖 Target Model 跑訓練集 40 題
                  ├──> [2. 分流與切片] 分成成功與失敗池，每 8 題切片
                  ├──> [3. 小批次分析] 🧠 Optimizer: analyst_error.md / analyst_success.md
                  ├──> [4. 階層式合併] 🧠 Optimizer: merge_failure.md -> merge_success.md -> merge_final.md
                  ├──> [5. 打分與排序] 🧠 Optimizer: ranking.md (裁剪至預算 Lt)
                  ├──> [6. 程式套用] 產生候選技能 s_tilde
                  └──> [7. 驗證守門員] 🤖 Target Model 跑驗證集 D_sel
                          ├──> [通過 (> score_cur)] 更新 s_cur 與 s_best
                          └──> [失敗 (<= score_cur)] 將 edits 與錯誤模式寫入 B
                  │
          └───> 【Epoch 結束】 (若 e >= 2)
                  ├──> [8. 慢速更新] 🧠 Optimizer: slow_update.md ──> 🤖 Target Model 跑驗證
                  └──> [9. 元技能更新] 🧠 Optimizer: meta_skill.md (不公開教練秘笈)
  │
  └───> 【訓練結束】
          └───> [10. 期末考] 🤖 Target Model 帶著 s_best 跑測試集 D_test ──> 輸出最終分數
```

---

### 7.2 巨細靡遺的執行步驟與 LLM 調用規格

#### 階段一：系統初始化 (Step 0)
1. **輸入參數載入**：讀取凍結目標模型 $M$、優化器模型 $O$、測試環境 $h$、資料集分割 $D_{tr}, D_{sel}, D_{test}$、初始技能 $s_0$、訓練週期 $E$、單步編輯預算排程 $L_t$、批次大小 $B$ 與小批次大小 $B_m$。
2. **狀態初始化**：
   * 當前技能 $s_{cur} \leftarrow s_0$
   * 歷史最佳技能 $s_{best} \leftarrow s_0$
   * 分數快取 $\mathcal{C} \leftarrow \emptyset$ (雜湊快取，防止重複執行相同技能的驗證)
   * 被拒修改緩衝區 $\mathcal{B} \leftarrow [ ]$
   * 優化器元技能記憶 $m_{meta} \leftarrow \emptyset$
3. **計算基準分數**：
   * 🤖 **[LLM 調用 - 目標模型 $M$]**：執行 $score_{cur} \leftarrow \text{Evaluate}(M, h, s_0, D_{sel})$。
   * 將分數寫入快取：$\mathcal{C}[\text{Hash}(s_0)] \leftarrow score_{cur}$，並設定歷史最佳分數 $score_{best} \leftarrow score_{cur}$。

---

#### 階段二：Epoch 外部循環 (Step 1)
4. 對於每個 Epoch $e = 1, 2, \dots, E$：
   * **洗牌**：將訓練集 $D_{tr}$ 隨機洗牌，劃分為多個大小為 $B = 40$ 的訓練批次。
   * **清空短期記憶**：清空被拒修改緩衝區 $\mathcal{B} \leftarrow [ ]$。

---

#### 階段三：Step 內部循環 (Step 2)
5. 對於當前 Epoch 內的每一個訓練批次（Step）：
   * **5.1 前向 Rollout 收集證據**：
     * 🤖 **[LLM 調用 - 目標模型 $M$]**：在當前訓練批次的 40 個任務上，帶著 $s_{cur}$ 執行任務，收集 40 份執行軌跡（Traces）。
   * **5.2 分流與切片**：
     * 將軌跡分類為成功池與失敗池。
     * 依照 $B_m = 8$ 的規格，將成功池與失敗池切片為多個 Minibatches。
   * **5.3 🧠 [LLM 調用 - 優化器模型 $O$]：小批次反向診斷**
     * 針對每個失敗 Minibatch，並行呼叫優化器。
       👉 *System Prompt 文件：* **`analyst_error.md`**
       * 註：若此時 $\mathcal{B}$ 與 $m_{meta}$ 有內容，會在此處作為 Context 注入。
     * 針對每個成功 Minibatch，並行呼叫優化器。
       👉 *System Prompt 文件：* **`analyst_success.md`**
       * 註：若此時 $m_{meta}$ 有內容，會在此處作為 Context 注入。
   * **5.4 🧠 [LLM 調用 - 優化器模型 $O$]：階層式合併**
     * 若失敗草案 $> 8$ 個，執行樹狀合併（Tree Reduce）。將失敗草案融合成 1 份 Unified Failure Patch。
       👉 *System Prompt 文件：* **`merge_failure.md`**
     * 若成功草案 $> 8$ 個，執行樹狀合併。將成功草案融合成 1 份 Unified Success Patch。
       👉 *System Prompt 文件：* **`merge_success.md`**
     * 執行最終跨堆決選合併（失敗修復優先，傳遞 `support_count`）。
       👉 *System Prompt 文件：* **`merge_final.md`**
   * **5.5 🧠 [LLM 調用 - 優化器模型 $O$]：排名與剪裁**
     * 根據四大優先度標準（系統影響力優先），從最終合併清單中，篩選出符合本回合編輯預算 $L_t$ 的候選 Edits。
       👉 *System Prompt 文件：* **`ranking.md`**
   * **5.6 程式套用產生候選人**：
     * 系統程式執行 Exact Match 字串操作（套用 append/replace 等），產生候選技能 $\tilde{s}$。若部分修改匹配失敗則 `skip`，並寫入 `edit_apply_report.json`。
   * **5.7 驗證集判定與回滾**：
     * 若 $\text{Hash}(\tilde{s})$ 存在於快取 $\mathcal{C}$，直接讀取分數 $score_{cand}$。
     * 否則，🤖 **[LLM 調用 - 目標模型 $M$]**：執行 $score_{cand} \leftarrow \text{Evaluate}(M, h, \tilde{s}, D_{sel})$。並寫入快取。
     * **判定分支**：
       * **若 $score_{cand} > score_{cur}$** (嚴格大於，接受新技能)：
         * 更新當前技能：$s_{cur} \leftarrow \tilde{s}$，當前分數：$score_{cur} \leftarrow score_{cand}$。
         * 若突破歷史最高分（$score_{cand} > score_{best}$）：更新 $s_{best} \leftarrow \tilde{s}$，$score_{best} \leftarrow score_{cand}$。
       * **若 $score_{cand} \le score_{cur}$** (拒絕，系統自動 Rollback 回滾)：
         * 維持原本技能：$s_{cur} \leftarrow s_{cur}$。
         * 記錄失敗經驗：將本次試圖修改的 edits 內容、未解決的失敗模式與分數變化量，寫入短期緩衝區 $\mathcal{B}$。

---

#### 階段四：Epoch 邊界宏觀調控 (Step 3) (僅在 $e \ge 2$ 時執行)
6. **🧠 [LLM 調用 - 優化器模型 $O$]：慢速更新生成 (Slow Update)**：
   * 系統對照 20 個隨機抽樣的相同任務在新舊技能下的對比成績單。
   * 優化器讀取對比成績單，寫出宏觀戰略，覆寫 `best_skill.md` 內的保護區（`<!-- SLOW_UPDATE -->`）。
     👉 *System Prompt 文件：* **`slow_update.md`**
   * 🤖 **[LLM 調用 - 目標模型 $M$]**：將加入新戰略的技能送往驗證集跑分。若分數變低，則拒絕並丟棄該戰略（回滾保護區文字）。
7. **🧠 [LLM 調用 - 優化器模型 $O$]：教練專屬元技能更新 (Meta Update)**：
   * 優化器總結這個 Epoch 的優化策略與避雷指南，更新元技能 $m_{meta}$（儲存於後台變數，僅在下一個 Epoch 作為 Optimizer 呼叫的 Context）。
     👉 *System Prompt 文件：* **`meta_skill.md`**

---

#### 階段五：最終測試 (Step 4)
8. **🤖 [LLM 調用 - 目標模型 $M$]：最終期末考**：
   * 訓練結束。提取在驗證集上取得最高分的歷史最佳技能 $s_{best}$。
   * 目標模型帶著 $s_{best}$，在從未見過的測試集上執行任務：
     $$score_{test} \leftarrow \text{Evaluate}(M, h, s_{best}, D_{test})$$
   * 輸出最終的測試分數與技能檔案 `best_skill.md`。

## 九、 結論與軟體工程實務反思 (Conclusion & Reflections)

本章節總結 SkillOpt 的核心技術貢獻，並以第一線 AI 軟體工程師的視角，客觀剖析該框架在實際生產環境落地時所面臨的工程摩擦力與限制。

---

### 9.1 本篇論文的核心技術貢獻總結

SkillOpt 的核心價值，在於**將「提示詞工程（Prompt Engineering）」從黑盒子的試錯（Trial-and-error），轉化為可預測、可量化且具備數學約束的「系統工程優化（Optimization Pipeline）」**。

透過將技能文件（`best_skill.md`）視為外部可變參數，系統成功映射了深度學習的關鍵組件：
*   **小批次梯度下降** $\rightarrow$ 並行小批次軌跡分析（`analyst_error.md`）。
*   **學習率與梯度裁剪** $\rightarrow$ Cosine Schedule 的編輯預算控制。
*   **驗證與回滾機制** $\rightarrow$ 獨立驗證集（$D_{sel}$）與嚴格的平手淘汰制。
*   **長短期記憶** $\rightarrow$ 被拒修改緩衝區（短期）與元技能（長期）的隔離儲存。

這種「離線訓練、導出部署」的架構，使 Agent 能夠在不更新任何權重的前提下大幅提升性能，且在正式部署時**增加 0 額外推理算力與延遲**。

---

### 9.2 實務落地之工程挑戰與反思

雖然 SkillOpt 在學術基準測試上取得了 52/52 的成績，但在實際軟體工程落地時，開發團隊必須評估以下三個核心痛點：

#### 1. 對「自動且精確的評估器（Evaluator）」的高度依賴
*   **挑戰**：Validation Gate（驗證守門員）是整個系統的靈魂。這意味著該任務**必須擁有低成本、高可靠且可自動執行的打分機制**。
*   **限制**：在程式執行、數值計算或有固定格式的任務中，這個設計運作得非常順暢。但若應用於寫作、創意生成、開放式客服等主觀性極強、難以用程式自動判斷對錯的場景，驗證集的自動化打分會面臨巨大挑戰（甚至可能需要引入昂貴的 LLM-as-a-judge 判定，進一步抬高訓練成本）。

#### 2. 離線訓練成本與長期部署效益的權衡 (Token ROI)
*   **挑戰**：雖然線上部署成本為 0，但離線優化過程極度消耗 Token。
*   **限制**：從 `Table 6` 可以看出，光是訓練一個數學或試算表技能，就需要跑 20M 到 200M 的 Token 運算。
*   **工程建議**：本框架最適合部署在**「高頻率重複運行」、「流程邏輯相對固定」且「錯誤代價極高」**的生產級 Agent 系統（如自動化財務報表處理、系統運維腳本執行）。對於一次性或低頻的簡單任務，這套優化管線的 Token 投資報酬率（ROI）可能並不划算。

#### 3. 單一技能文件的表達力瓶頸 (Single-file Bottleneck)
*   **挑戰**：SkillOpt 刻意將優化對象限制在單一的 `best_skill.md` 檔案，以維持簡潔性。
*   **限制**：在面對極度異質、包含數百種不同業務場景的超大型企業級環境中，單一 Markdown 檔案很快就會達到 context 限制，且不同場景的規則容易在同一個檔案內產生邏輯互斥。
*   **未來演進方向**：工程團隊未來若要擴展此系統，可能必須將其與「技能庫路由（Skill Library Routing）」結合——即使用 SkillOpt 分別優化多個子領域的 md 文件，並在運行時透過調度器進行路由，以突破單一檔案的表達力上限。
