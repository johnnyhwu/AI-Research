# 深度解析 Claude Code：一套生產級 AI Agent 的系統工程實踐

## 前言

如果你打開 Claude Code 的原始碼庫，會看到超過五十萬行 TypeScript。但根據對這套系統的逆向分析，真正屬於「AI 決策邏輯」的程式碼只佔了其中的 1.6%，剩下 98.4% 全是維運基礎設施與防護機制。

這個數字很值得停下來想一想。多數人以為 Agent 的能力來自更聰明的提示詞，或是更龐大的工具清單。但 Claude Code 的架構告訴我們一個不太一樣的故事：模型只是一個無狀態的推理引擎，真正撐起整套系統可靠運作的，是它外面那層厚厚的工程防護網，也就是所謂的 Harness。

這篇文章會帶你從核心迴圈開始，一路看到安全防禦、記憶管理、擴充機制、狀態持久化，最後談到 Claude Code 跟另一種架構取向（常駐型網關 OpenClaw）的對比，以及它在商業防禦與工程文化上留下的痕跡。目標是建立一個可以套用在你自己專案上的心智模型：設計 Agent 系統，重點從來不是讓模型更聰明，而是怎麼幫它把防護服穿好、把工作台佈置好。

## 一、模型只是引擎，Harness 才是那台車

Claude Code 的定位是一個短暫存活的 CLI 本地代理人。它綁定在單一專案目錄底下，擁有讀寫檔案、執行 shell 指令的權限，生命週期跟著終端機開關生滅。這跟 GitHub Copilot 那種行內自動補全工具完全不同，它是真正具備代理能力的工具。

支撐這套架構的，是五個核心價值觀，而且這五個價值觀在後面每一個子系統裡都留下了明顯的痕跡：

- **人類決策權**：系統的終極控制權要留在人類手上。防護網預設攔截未知指令（deny-first），遇到不確定的操作就跳出授權請求，而不是讓 AI 默默把危險指令跑完。
- **安全、資安與隱私**：安全是系統的義務，不能只靠人類警覺。因為人類核准權限請求的機率高達 93%，「授權疲勞」讓人類其實不是可靠的守門員，所以還需要獨立的 ML 安全分類器與作業系統級沙盒。
- **可靠執行**：上下文視窗是系統最稀缺的資源，要能撐過極長的對話而不崩潰，這也是後面五層壓縮管線誕生的原因。
- **能力放大**：系統要能處理「沒有 AI 時工程師根本不想碰」的繁瑣任務。做法是最小化邏輯鷹架，不用寫死的分支或狀態機限制模型，而是給模型盡量大的工具箱與行動自由。
- **上下文適應性**：系統要能配合每個人的專案環境與習慣，這就是 Hooks、Skills、Plugins、MCP（Model Context Protocol）四種擴充機制，加上透明的 `CLAUDE.md` 記憶階層存在的理由。

把這五個價值觀落地成架構，可以畫成一條資料流：使用者與介面（CLI/SDK）在最外層，中間是 Agent 迴圈（`queryLoop`）搭配狀態與持久化模組，外面包一層權限系統把關，最底層才是實際執行的工具池與沙盒環境。整條路徑上，每一個環節都在服務同一個設計原則：給模型最大的決策自由，但把落地成行動的權力，牢牢握在防護網手裡。

## 二、迴圈是心臟：queryLoop 怎麼一步步消除不確定性

理解架構的骨架之後，更重要的是搞懂 Agent 為什麼能表現得像是「什麼都會」。答案不是它內建了一百種針對不同任務的演算法，而是那句常被引用的話：通用的能力不在工具本身，而在那個迴圈。

Agent 的每一次工具呼叫，本質上都是一次不確定性遞減。舉個例子：當使用者說「幫我修復 `auth.py` 裡導致登入失敗的 bug」，模型一開始其實什麼都不知道。它會透過一連串小決策，把不確定性一步步壓下去：先讀 `auth.py` 看到資料庫連線相關的程式碼，不確定性從八成五降到六成五；接著用 grep 找到專案裡其他用資料庫的地方，搞懂 coding style，降到四成；再跑一次測試，看到具體的 error stack trace（比如某一行的型別不匹配），降到兩成；最後透過 LSP 拿到編譯器的診斷結果，確認錯誤根源，降到不到一成，然後才真正動手修改。模型不需要一次就猜對答案，它只需要每一輪都挑出最能消除當下最大不確定性的那個工具。

這套心智模型背後，是一個嚴格定義的九步生命週期，每一個 agentic turn 都要走過：解析設定（system prompt、工具清單、安全性設定）、初始化一個乾淨的狀態物件（採用整物件覆寫而非局部修改，確保狀態一致）、組裝上下文（對話歷史、`CLAUDE.md`、自動記憶）、跑一次前置壓縮整形、串流呼叫模型、解析回傳的工具呼叫並派發、通過權限閘門、實際執行並收集結果、最後判斷是否終止。這九步不是各自獨立的模組，而是一條被嚴格管控的管線，確保模型每次要採取行動之前，看到的環境都是最新、最準確的狀態。

當模型一次丟出多個工具請求時（比如同時讀三個檔案），系統會依照工具性質分流：唯讀操作屬於「並發安全」，可以平行跑來省時間；會改變狀態的操作（跑 shell 寫入指令、改檔案）屬於「排他執行」，必須排隊依序跑，避免競態條件把程式碼寫爛。這裡還有一個很實用的設計叫「手足中止」：如果同時發出的三個 bash 指令裡有一個報錯，系統會立刻觸發中止訊號，把還在跑的另外兩個直接砍掉，避免浪費運算資源、避免錯誤繼續擴散。

遇到極端狀況時，系統也有三套自癒機制：輸出被 `max_tokens` 卡住時，自動調高上限重試最多三次；API 回報 prompt 太長時，直接跳過輕量壓縮、上最強力的壓縮把上下文瘦身後重送；主力模型出問題時，自動降級用較小的模型維持運作。這幾個機制合起來的意思是，一個好的 Harness 不能只在順利的路徑上工作，它得先假設錯誤一定會發生，然後把復原邏輯內建進迴圈裡。

## 三、七層防護網：讓安全不依賴人類的警覺性

AI Agent 最危險的地方，就是它真的能執行 shell 指令、真的能改檔案。而前面提過的那個數字——使用者核准權限請求的機率高達九成三——講白了就是：人類靠不住。所以 Claude Code 的核心設計是，權限控制必須由 Harness 獨立執行，不管使用者這個當下有沒有在看螢幕。

具體做法是一個七層過濾漏斗，模型發出的每一個工具呼叫請求都得依序穿過去：

1. **工具預先過濾**：送出 prompt 前，先把禁用的工具整個剔除。
2. **拒絕規則引擎**：deny 規則永遠優先於 allow。
3. **hook 攔截**：讓專案自訂的程式化審核介入。
4. **權限模式約束**：根據目前的權限模式決定要不要跳出確認視窗。
5. **ML 分類器**：交給一個獨立的小模型做意圖安全性分析。
6. **作業系統沙盒**：最後一層物理隔離，限制檔案與網路存取。
7. **會話邊界歸零**：每次跨會話重啟後，所有先前授權過的權限全部歸零。

權限模式本身也是一條漸進式的信任光譜，從最保守到最放任分成七種：

| 模式 | 自主性 | 行為 |
|---|---|---|
| `plan` | 極低 | AI 只能提案，連讀檔都要人類手動確認 |
| `default` | 低 | 預設模式，大部分有副作用的操作都要跳窗確認 |
| `acceptEdits` | 中 | 允許直接改檔案，但執行指令仍要確認 |
| `auto` | 中高 | 啟動 ML 分類器，判斷無害就自動放行 |
| `dontAsk` | 高 | 不再跳確認視窗，但寫死的 deny 規則依然生效 |
| `bypassPermissions` | 極高 | 幾乎跳過所有檢查，只留下核心安全硬限制 |
| `bubble`（內部專用） | 特殊 | 子代理把權限請求往上冒泡，回傳給母會話決定 |

設計這個光譜時考慮的是動作的可逆性：改程式碼可以用 git 復原，可逆性高，所以 `acceptEdits` 敢放行；但跑 `npm install` 可能拉到惡意套件，不可逆，所以得更嚴格。

這裡有個設計特別值得學：權限被拒絕不會讓系統崩潰，而是變成一個路由訊號。假設模型想跑 `rm -rf /` 清空間，deny-first 引擎攔下來之後，系統不是直接報錯，而是回傳一段結果告訴模型「這個操作有危險性，已被安全策略擋下，請改用更安全的方法或說明你的意圖」。模型收到之後會自我反思，可能改用逐一檢查檔案再刪除的方式繼續任務。換句話說，Harness 扮演的角色不是警察，比較像是帶著邊界的教練，用拒絕來引導模型走向更專業的工程路徑，而不是單純攔下來就結束。

老實說，這套防護網也不是無懈可擊。有個很現實的限制是，如果一個 shell 指令包含超過五十個子指令，系統為了避免解析開銷拖垮 UI，會自動跳過詳細的規則比對，退回到單一的通用詢問——換句話說，效能壓力可以讓安全性打折扣。另一個更麻煩的問題是初始化時序漏洞：專案初始化階段要載入 plugin 跟 hook，這個動作是在權限對話框出現之前就執行的，等於留了一個時間窗，惡意程式碼可以趁保護機制還沒啟動前先完成攻擊。再加上使用者對系統的信任會隨時間習慣化，一旦人類養成看到彈窗就按確認的反射動作，前面四層過濾其實形同虛設，最後真正撐住場面的只剩 ML 分類器跟沙盒隔離這兩層。

## 四、上下文是最貴的記憶體：分層載入與五級壓縮

在 Agent 系統裡，上下文視窗是最稀缺也最貴的資源。Claude Code 雖然支援到 200K 甚至 1M 的長度，但頻繁讀檔跟報錯訊息可以瞬間把容量塞滿。為了應付這件事，Harness 實作了一套很像作業系統記憶體管理的分層載入機制。

呼叫模型前，系統會依照不同的刷新頻率把資訊拼進上下文：系統提示詞跟工具指令集是整個會話快取一次；`CLAUDE.md` 的四層規則跟 git 基本狀態也是快取一次；自動記憶則是每一回合動態注入，靠一個小模型預先挑選相關的專案知識；對話歷史跟執行結果（讀到的程式碼、bash 輸出、子代理總結）是持續累積；完整的工具說明書則是按需觸發，只有模型主動呼叫 `ToolSearch` 之後才載入。

靜態記憶的部分，Claude Code 刻意不用向量資料庫，而是採用純文字的 `CLAUDE.md` 檔案，確保開發者對 AI 記憶有百分之百的審計權跟修改權。載入順序是由遠到近：系統級規範、使用者個人習慣、專案共用規則、最後是工作目錄裡的私有設定，而且越靠近工作目錄的檔案越晚載入。這是刻意利用 LLM 對近因的偏好——越晚出現的資訊，模型的注意力權重越高。

動態記憶的部分則靠兩套機制撐著：自動記憶是背景發一個非同步請求，讓小模型快速掃過所有記憶檔案的標題，挑出跟目前提示詞最相關的一到五份文件，在送出前的最後一刻注入；工具說明書延遲加載則是一開始只給模型看工具名單，等模型真的呼叫 `ToolSearch` 想了解某個 MCP 工具細節，才把幾千字的完整說明書載進來。這避免了把模型根本用不到的工具塞滿上下文。

當 token 壓力真的逼近極限，系統會啟動一套五層壓縮管線，由便宜到昂貴、由物理性到語意性依序執行：

| 層級 | 名稱 | 做法 | 舉例 |
|---|---|---|---|
| L1 | 預算縮減 | 截斷單一過長輸出 | 把上萬行的錯誤 log 縮成首尾各 500 行 |
| L2 | 裁切 | 暴力移除最舊對話 | 直接丟掉一開始的前十個回合 |
| L3 | 微壓縮 | 結構化 metadata 微縮 | 快取感知優化，縮減 id 但不破壞 prefix |
| L4 | 上下文折疊 | 建立虛擬視圖 | 用摘要暫時取代原文，原文仍留在磁碟上 |
| L5 | 自動總結 | 呼叫模型重寫摘要 | 把舊記憶徹底改寫成一段精華摘要 |

最精妙的地方是 L3 怎麼在不破壞 prompt caching 的前提下擠出空間。只要壓縮動了一個字，prefix 就會變，快取就會 miss，系統變慢也變貴。解法是「先發車、後補票」：發現壓力時先忍住不改，原封不動送出這一輪，等成功打中快取之後，再讀取 API 回傳 header 裡的 `cache_deleted_input_tokens` 收據，確認伺服器精確丟棄了多少 token，這時候才在本地建立壓縮邊界跟對應的 UUID 鏈路，讓下一輪能以最精準的帳本重新對齊快取。萬一在這個過程中真的撞牆，API 回報 prompt 太長，系統就會觸發反應式重試，直接上 L5 強制壓縮再重送。

## 五、擴充四式與召喚分身

一個生產級的 Agent 不能只會寫程式，還得跟資料庫、Slack、搜尋引擎這些外部世界溝通。Claude Code 的想法是：不同的擴充功能，應該根據它對上下文造成的負擔，採用不同的接入機制。

具體分成四類，插在迴圈的不同生命週期裡：

| 機制 | 核心能力 | 上下文成本 | 插入點 |
|---|---|---|---|
| Hooks | 生命週期攔截、事件自動化 | 零 | 工具執行前後 |
| Skills | 特定領域的指令集與複合動作 | 低 | 上下文組裝階段 |
| Plugins | 多組件打包分發的容器 | 中 | 橫跨迴圈所有插入點 |
| MCP | 外部服務整合 | 高 | 工具池組裝階段 |

Hooks 是系統的神經反射，直接寫死在 Harness 裡，事件觸發就執行，模型完全感覺不到，token 成本是零。實務上像 `PreToolUse` 這種 hook，可以偷偷改寫模型想跑的危險指令，或是叫一個小模型做安全審查。Skills 則是模型的專業手冊，每個 skill 是一份 `.md` 檔案，一開始只給模型看名稱加一句話描述，模型真的想用才載入數千字的完整內容，這就是延遲技能發現。MCP 是標準的 JSON-RPC 協定，讓 Claude 接上 Google Search 或 Postgres 這類外部服務，同樣支援延遲工具定義，模型要用才透過 `ToolSearch` 拿說明書。Plugin 本身不是一個功能，而是一個包裝盒，一份 manifest 可以同時打包斜線指令、自訂代理人、技能組、事件監聽器、MCP/LSP 伺服器，甚至 UI 樣式，讓第三方開發者一次性給 Agent 裝上新的眼睛、新的手跟新的反射。

另外值得一提的是 LSP（Language Server Protocol）的整合。Claude Code 之所以比一般 chatbot 更懂程式碼，是因為它不靠 grep 瞎找，而是呼叫 `go_to_definition` 精準跳到函式定義；改完程式碼會立刻呼叫 `get_diagnostics` 問編譯器有沒有錯；也不用讀整個檔案，呼叫 `hover` 拿函式的型別簽名就夠了。這一整套讓模型省下大量原本要花在盲目搜尋上的 token。

當任務牽涉多個檔案或高風險實驗時，母代理會呼叫 `AgentTool` 生出子代理，這裡有三個嚴謹的設計：第一是物理級隔離，子代理在獨立的 git worktree 裡工作，不管它把程式碼改得多爛、測試跑得多崩，人類在主視窗看到的原始碼都是安全的；第二是權限向上冒泡，子代理想跑危險指令時會觸發前面提過的 `bubble` 模式，權限請求傳回母會話，最後還是由人類點頭；第三是僅限總結回傳，子代理可能在自己的上下文裡打了一場五萬 token 的仗，但結束後會把所有歷史燒掉，只回傳一份大概三百字的報告給母代理。這麼做是因為子代理的 token 消耗量通常是單人操作的七倍，如果把完整過程都塞回母代理的上下文，母代理很快就會被噪音撐爆。

## 六、只增不改的日誌：狀態怎麼存、怎麼恢復

持久化不只是為了存檔，更是為了讓系統在崩潰或被壓縮之後，還能重建出完整的狀態。Claude Code 放棄了傳統的關聯式資料庫，改用一種只增不改（append-only）的 JSONL 日誌哲學。

好處很直接：每一行都是獨立的 JSON 物件，開發者可以直接用 `tail`、`grep` 或任何文字編輯器打開對話紀錄，不需要專門的資料庫檢視工具；而且逐行記錄的格式對 git 很友善，diff 起來一目瞭然。系統維護三條獨立的資料通道：專案目錄下的會話劇本（`[sessionId].jsonl`，存放單一 session 的完整細節）、使用者家目錄下的全域提示詞歷史（`history.jsonl`，只存 prompt，給 CLI 的 `Ctrl+R` 搜尋用）、以及子代理側鏈（`[subagentId].jsonl`，獨立存放子代理的過程，避免污染主對話紀錄）。

這裡有個常被忽略但很關鍵的細節：紀錄檔裡不能只存對話內容，必須把模型看到的一切都存下來，包括每一筆工具執行結果。假設 AI 曾經讀過 `config.ts` 這五百行程式碼，並根據內容改了東西，隔天你用 `--resume` 恢復對話時，模型的大腦其實是空的。如果 JSONL 裡沒有存那五百行，模型根本不知道自己昨天為什麼要那樣改，很容易產生幻覺。如果工具回傳的內容太長（比如一份很長的 log），系統會存一個內容參照標籤，恢復時再依標籤找回原始資料。

那被壓縮掉的舊歷史，恢復時要怎麼處理？系統會在 JSONL 裡插入一個壓縮邊界事件，帶著三種 UUID：`headUuid` 指向被壓縮段落的開頭，`anchorUuid` 把摘要錨定在正確的時間點，`tailUuid` 指向壓縮段落的結尾。載入檔案的時候，系統讀到摘要物件、看到這些 UUID 指標，就會自動跳過磁碟上已經標記為壓縮的原始訊息，在記憶體裡重新拼接訊息之間的關聯。結果是，雖然磁碟上的檔案是碎裂的，送進模型的卻是一條邏輯完整連貫的記憶長鏈。

有一個設計乍看反直覺，但其實很合理：恢復會話時，AI 會記得所有聊天內容，但系統會忘掉你之前按過的每一個 allow/deny 授權。這叫安全不變性——每一個 session 都被視為獨立的信任網域，昨天安全的環境不代表今天也安全（專案可能在半夜被植入惡意指令）。代價是使用者得對同一個危險動作重新點一次確認，換來的好處是每一項敏感操作的授權，都是基於當下環境做出的清醒決策，不會被過期的權限鑽漏洞。除了對話紀錄，系統也對檔案系統做了保護：`fork` 可以複製 JSONL，建立一個平行的對話空間，實驗失敗還能隨時切回原本分支；`rewind-files` 則靠 `~/.claude/file-history/` 底下存的檔案快照，把程式碼還原到特定的對話時間點。

## 七、換個場景，架構就整個翻轉

把 Claude Code 拿去跟另一種截然不同的系統 OpenClaw 對比，可以看到部署情境怎麼決定架構長相。Claude Code 像單幹的私家偵探，專注在單一專案上；OpenClaw 更像一間大型跨國飯店的總機，身兼多渠道的個人助理網關。

| 維度 | Claude Code（偵探型） | OpenClaw（飯店型） |
|---|---|---|
| 系統角色 | 專注單一專案的 coding harness | 多渠道個人助理網關 |
| 生命週期 | 短暫，跟著 CLI 開關生滅 | 常駐，以 daemon 形式運作 |
| 信任模型 | 不信任模型本身 | 不信任外部輸入 |
| 安全邊界 | 動作級別，七層漏斗 | 周邊級別，白名單與配對 |
| 記憶管理 | 專案 Markdown + 五層壓縮 | 結構化長期記憶 + 背景整理機制 |

兩者最大的結構差異在於誰是老大。Claude Code 是以迴圈為中心，`queryLoop` 是宇宙中心，所有 UI 顯示、工具執行、權限檢查都是為了服務這個迴圈，適合處理需要長路徑推理的重度任務。OpenClaw 反過來，是以 RPC 分發層為中心，網關才是老大，agent 迴圈只是被叫用的一個插件，因為網關要同時處理來自不同通訊軟體的訊息，得先排隊、辨識身分、解析 session，確認沒問題才把工作發包給內部的 agent 迴圈。這也反映在記憶哲學上：Claude Code 為了省錢，把力氣花在壓縮技術上；OpenClaw 為了長期助理的需求，發展出把瑣事提煉成永久知識的機制，寫進長期記憶檔案。

這兩套架構並不是競爭關係，而是可以互相嵌套的俄羅斯娃娃。透過一種叫 ACP（Agent Client Protocol）的協定，你可以讓 OpenClaw 當前端管家，接收你在通訊軟體上下的指令，再由它啟動一個 Claude Code 執行緒當作內部子系統去真正動手做事。未來的 Agent 生態很可能不會只有一種架構獨大，而是網關級系統（管溝通跟記憶）包著任務級框架（管執行跟安全）這樣一層一層疊起來。

## 八、看不見的攻防與工程文化

除了技術架構，Claude Code 的原始碼裡還留下不少商業防禦的痕跡，這部分挺有意思，值得單獨拉出來談。

AI 領域最大的威脅不是程式碼被偷，而是模型智慧被蒸餾——競爭對手用便宜的模型大量抓取輸入輸出，拿去訓練自己的模型。針對這件事，Claude Code 在發送 API 請求時做了兩手準備：一是在系統提示詞的工具清單裡隨機塞進幾個根本不存在的假工具，對手如果拿這些資料去訓練，他們的模型會學到錯誤知識，未來嘗試呼叫這些不存在的工具時就會出包或產生幻覺；二是把最值錢的推理過程在伺服器端過濾或加密處理，確保就算客戶端程式碼被破解，也拿不到那套解題思路。

另一個有意思的設計是跨語言的授權驗證。TypeScript 像玻璃屋一樣透明，很容易被 monkey-patch 破解，比如直接把 `if(license)` 改成 `if(true)`。為了對付這個，系統的做法是：TS 原始碼先送出一個誘餌字串，駭客以為模擬這個字串就能矇混過關；但在請求真正離開電腦前，底層用 Zig 寫成、編譯成二進位機器碼的執行引擎（Bun）會介入，用預先編譯好的演算法算出真正的加密雜湊值，直接覆寫掉那個誘餌標籤。這麼一來，要破解這套認證，對手得去撬開 Zig 編譯出來的那個保險箱，攻擊成本一下拉高好幾個量級。

系統裡也明確定義了哪些提示詞內容是可以快取的靜態規則、哪些是不能快取的動態資訊（比如當下時間、git 分支），避免開發團隊不小心把常變動的變數塞進靜態區，拖累快取命中率、拉高 API 費用。還有一個叫臥底模式的機制：當 Anthropic 員工用 Claude Code 參與外部開源專案時，這個模式會強制啟動且不能關閉，自動抹掉所有內部模型代號，也絕不承認內容是 AI 生成的。這既是商業保護，某種程度上也算一種特殊的網路倫理實踐。

工程文化這邊也留下了痕跡：工程師在 Markdown 文件開頭加上特定標籤，系統閒置時就會自動喚醒一個背景子代理，根據最近的程式碼變更重寫並更新文件內容，算是實現了文件跟程式碼同步這件事的一種解法。原始碼裡也出現不少「寫給 AI 看」的註解，說明程式碼的讀者已經不只是人類，開發者開始有意識地引導 AI 該怎麼理解一段複雜的非同步邏輯。

## 九、下一步：從被動工具走向主動夥伴，以及還沒解決的問題

目前的 Claude Code，包括大多數 Agent，基本上都是不點不亮的：使用者下指令，AI 才動作一次。但原始碼裡藏著一個叫 KAIROS 的架構雛形，展示了下一步可能的樣子——從 prompt 驅動轉向心跳驅動。使用者不在電腦前時，系統會定時注入一個心跳訊號，模型收到之後檢查專案有沒有掛起的任務（正在跑的編譯、待更新的文件），自主決定要不要動作。這種主動型 agent 最大的問題是 token 會燒完，所以引入了一個 `SleepTool`：模型判斷暫時沒事做，就呼叫它讓系統進入休眠，暫停心跳，直到快取快過期或使用者回來為止。

還有一個很嚴肅的提醒：百分之七十八的 AI 失敗是無聲的。AI 不會報錯，它只是自信地寫出有 bug 的程式碼。解法的方向是讓生成跟評估分開：一個角色負責思考跟寫程式（generator），一個角色負責審查輸出是否符合規範（evaluator），還有一個角色用編譯器、LSP、單元測試這類確定性工具提供最後的真理（verifier）。設計系統時不能只優化提示詞，要設計一個生成者跟評估者互相對抗的工作流，讓失敗在進到使用者視線前就被攔下來。

最後，也是整個架構裡最需要正視的部分，是監督悖論：Agent 越強大，對人類長期的影響可能反而越麻煩。短期內生產力確實會爆發，但長期下來專案複雜度可能跟著飆升。根源有兩個，一個是認知卸載——人類因為信任 AI，慢慢不再檢查細節，對系統失去掌控；另一個是上下文碎片化——因為壓縮管線的存在，AI 看到的永遠只是專案的片段，容易寫出重複或互相衝突的邏輯。目前包括 Claude Code 在內，大多數 Agent 架構都還沒有內建解決這個問題的機制。未來的架構師要面對的課題，可能不只是怎麼讓 AI 更強，而是怎麼設計出一個能強迫人類持續學習、或是能維持專案全域透明度的架構。

## 結論

回頭看這整套架構，最重要的一課大概是：LLM 的能力決定了 Agent 的上限，但 Harness 決定了它的下限與穩定性。從心臟一樣的 `queryLoop`，到七層安全防護網、五層壓縮管線、四種擴充機制、只增不改的日誌哲學，再到藏在原始碼裡的商業防禦手法，這些拼起來才是一套真正能撐在生產環境裡運作的系統。

如果你也在設計自己的 Agent，這篇文章想留給你的重點是：不要只把力氣花在寫更好的 prompt 上，而是去思考怎麼設計一套為 AI 量身打造的作業系統——精算每一個 token 的去留，築起不依賴人類警覺性的防禦邊界，同時誠實面對目前架構還沒解決的問題，比如監督悖論。這些課題不會隨著模型變聰明就自動消失，反而會隨著 Agent 越來越強大變得更迫切。
