## 1. 導論與核心哲學 (Introduction & Core Philosophy)

在探討具體的 TypeScript 原始碼之前，我們必須先建立正確的「架構心智模型」。傳統的機器學習研究往往專注於如何訓練出更聰明的大型語言模型（LLM），但對於生產級（Production-grade）的 AI Agent 而言，**「擁有一顆聰明的大腦只是第一步，如何為這顆大腦穿上防護服、配備工作台，才是真正的工程挑戰」**。

本章節將以 Anthropic 的 Claude Code 為例，揭開真實商業環境下，AI Agent 系統的底層定位與設計哲學。

### 1.1 系統定位與 5 大核心價值觀

**系統定位：短暫的 CLI 本地代理人 (Ephemeral CLI Agent)**
與 GitHub Copilot 等「行內自動補全（Inline completion）」工具不同，Claude Code 是一個**完全具備代理能力（Agentic）**的終端機命令列（CLI）工具。它綁定於單一的本地專案目錄，具有讀寫檔案、執行 Shell 指令的權限，且生命週期隨著終端機的開關而生滅（Session-based）。

根據系統的原始碼與架構設計，Anthropic 在打造 Claude Code 時，底層是被以下 **5 大核心價值觀** 所驅動的，這 5 個價值觀在後續的每一個子系統中都留下了深深的烙印：

1. **人類決策權 (Human Decision Authority)**
   * **定義**：系統的終極控制權必須保留在人類手中。
   * **工程體現**：架構被設計為人類可以「隨時知情地介入」。例如：防護網預設會攔截未知的指令（預設拒絕，Deny-first），並向使用者發出授權請求，而不是讓 AI 默默地把危險程式碼跑完。
2. **安全、資安與隱私 (Safety, Security, and Privacy)**
   * **定義**：安全性是系統的「義務」，即使人類粗心大意，系統也要防範未然。
   * **工程體現**：人類有高達 93% 的機率會因為「授權疲勞（Approval fatigue）」而無腦點擊同意。因此，Harness（防護網）不能只依賴人類確認，必須實作獨立的機器學習安全分類器（Auto-mode Classifier）與作業系統級的隔離沙盒（Sandboxing）。
3. **可靠執行 (Reliable Execution)**
   * **定義**：確保系統能正確理解意圖，並在跨越極長的對話邊界（Context Boundaries）時依然保持連貫。
   * **工程體現**：LLM 的上下文視窗被視為系統「最稀缺的資源」。因此誕生了極度複雜的「五層漸進式壓縮管線（Compaction Pipeline）」來防止大腦容量溢出而崩潰。
4. **能力放大 (Capability Amplification)**
   * **定義**：系統必須能實質提升人類產出，處理那些「沒有 AI 時工程師根本不想嘗試」的繁瑣任務。
   * **工程體現**：最小化邏輯鷹架（Minimal Scaffolding）。系統不透過寫死的分支圖或狀態機來限制模型，而是給予模型在豐富工具箱中最大的行動自由度。
5. **上下文適應性 (Contextual Adaptability)**
   * **定義**：系統必須能動態適應每一位使用者獨特的專案環境、工具與習慣。
   * **工程體現**：實作了四種漸進式成本的擴展機制（MCP, Plugins, Skills, Hooks），以及純文字、透明的 `CLAUDE.md` 記憶階層系統。

> ⚠️ **資深架構師的警示：第六個隱藏視角 —— 監督悖論 (Paradox of Supervision)**
> 除了上述 5 點，我們在設計 Agent 時必須隨時戴上「長期能力保存」的評估透鏡。數據表明，過度依賴高度自動化的 Agent，初期會帶來 281% 的速度提升，但幾個月後卻會因為「人類喪失對專案的全域理解」，導致專案複雜度飆升 40.7%，進而產生嚴重的技術債。**目前多數 Agent 架構（包含 Claude Code）都還缺乏解決這個「能力退化問題」的內建機制。**

### 1.2 系統架構的高階視角：1.6% 決策與 98.4% 防護網

這篇逆向工程分析揭露了一個令人震撼的數據：**在 Claude Code 總計超過 50 萬行的 TypeScript 原始碼中，真正屬於「AI 決策邏輯」的程式碼大約只佔 1.6%，其餘的 98.4% 全都是運營基礎設施與防護網（Harness）。**

這確立了現代 Agent 開發的一個重要典範轉移（Paradigm Shift）：**LLM 只是系統裡的一個「無狀態推理引擎（Stateless Completion Endpoint）」，真正的作業系統（OS）是外面的那層 Harness。**

*[📝 圖片提示：請在此處插入論文中的 Figure 1 高階系統架構圖 (High-level system structure of Claude Code)]*

**7 大核心組件與 5 層子系統架構**
所有的外部輸入，最終都會匯聚到一條被嚴格管控的資料流中，這構成了系統的 7 大組件（可視為 5 個垂直的技術層級）：

```text
[ 表面層 Surface ] ─> (1) 使用者 (User) & (2) 介面 (CLI/SDK)
        │
[ 核心層 Core ] ────> (3) Agent 迴圈 (queryLoop) <─── (6) 狀態與持久化 (State/Memory)
        │
[ 防護層 Safety ] ──> (4) 權限系統 (Permission System)
        │
[ 執行層 Backend ] ─> (5) 工具池 (Tools / MCP) ───> (7) 執行環境 (Execution Env/Sandbox)
```

**設計原則：給予決策的自由，限制執行的權限**
在這種「大防護網」的架構下，Claude Code 選擇了 **「最少的鷹架，最大的運行防護網 (Minimal scaffolding, maximal operational harness)」**。這意味著：
* **模型端**：沒有死板的 LangGraph 節點或樹狀搜尋限制，模型想做什麼就做什麼，享有 100% 的推理自主權。
* **Harness 端**：所有的想法在落地為「行動」前，都會被 `Permission System` 攔截、審查、過濾。

### 1.3 Agent 的本質：不確定性遞減模型 (Uncertainty Reduction)

了解了架構骨架後，我們需要一個高級的「心智模型」來理解 Agent 為什麼能展現出看似「通用人工智慧（AGI）」的行為。

很多人誤以為一個通用型 Agent（Generalist Agent），是因為它的底層寫了一百種針對不同任務的演算法。但正如資深工程師所言：**"The generalism is not in the tools. It is in the loop."（通用的能力不在於工具本身，而在於那個迴圈。）**

Agent 的每一次 `Tool Use`，本質上都是一個**「不確定性遞減（Uncertainty Reduction）」**的過程。

**💡 實戰案例解析：修復未知的 Bug**
當使用者輸入：「幫我修復 `auth.py` 中導致登入失敗的 Bug」時，模型一開始其實是處於「極度無知」的狀態。這套單純的 Agent 迴圈會透過一系列微小的決策，一步步將不確定性降至零：

```text
[ 任務開始 ] 
 ❓ 不確定性：85% (不知道程式碼長怎樣，不知道報錯是什麼)

 ⬇️ Step 1: 呼叫 FileReadTool("auth.py")
 👁️ 模型看到了原始碼，發現有用到特定的資料庫連線。
 ❓ 不確定性降至：65%

 ⬇️ Step 2: 呼叫 GrepTool("Database connection")
 👁️ 模型找到了專案中其他使用資料庫的地方，理解了專案的 Coding Style。
 ❓ 不確定性降至：40%

 ⬇️ Step 3: 呼叫 BashTool("pytest test_auth.py")
 👁️ 模型看到了終端機噴出的具體 Error Stack Trace (如: Type mismatch on line 42)。
 ❓ 不確定性降至：20%

 ⬇️ Step 4: 呼叫 LSPTool.get_diagnostics()
 👁️ 透過編譯器語法樹確認了型別錯誤的根源。
 ❓ 不確定性降至：5%

 ⬇️ Step 5: 呼叫 FileEditTool("auth.py", 修正邏輯)
 🎯 成功完成任務！
```

**心智模型總結**：
Claude Code 的架構之所以強大，是因為它提供了一個極度穩定的 `queryLoop`（負責循環）和豐富的工具箱（負責消除盲點）。模型不需要「一次猜對（One-shot）」所有答案，它只需要在每一次迴圈中，**精準地選擇「能幫它消除當前最大不確定性」的那個工具**即可。

## 2. 系統心臟：代理人迴圈 (The Agentic Loop)

在 Claude Code 的實作中，所有的自主行為都源自於一個遵循 **ReAct (Reasoning + Acting)** 模式的 `while` 迴圈。這個迴圈並非簡單的 API 呼叫，而是一個包含了狀態持久化、動態壓縮、並發管理與錯誤恢復的複雜工程管線。

### 2.1 `queryLoop` 的 9 步標準生命週期

根據原始碼 `query.ts` 的定義，每一個 Agentic Turn（代理回合）都必須嚴格執行以下 9 個步驟。這套流程確保了模型在採取行動前，環境與上下文都是處於最穩定且最精確的狀態。

*[📝 圖片提示：請在此處插入論文中的 Figure 2 Runtime turn flow，展示回合與迴圈的交互關係]*

**單一回合的生命週期分解：**

1.  **設定解析 (Settings Resolution)**：解構並載入 System Prompt、工具說明書清單、以及目前的安全性設定。
2.  **狀態初始化 (Mutable State Initialization)**：建立一個單一的 `State` 物件，用來暫存本回合的所有動態數據。Harness 堅持使用「全物件覆寫」而非局部修改，以確保狀態的一致性。
3.  **上下文組裝 (Context Assembly)**：從硬碟與記憶體中抓取對話歷史、專案規則（`CLAUDE.md`）與自動記憶（Auto Memory）。
4.  **前置上下文整形 (Pre-model context shapers)**：在呼叫模型前的最後一刻，執行 5 層壓縮管線（如 L1-L3），確保剩餘空間足以容納下一個動作。
5.  **模型呼叫 (Model Call)**：透過 `for await` 迴圈開啟串流呼叫（Streaming）。模型開始產出 Reasoning（思考）與 `tool_use`（工具呼叫）。
6.  **工具派發 (Tool-use dispatch)**：解析模型傳回的 `<tool_use>` 標籤，並將其交給工具調度層（Tool Orchestration）。
7.  **權限閘門 (Permission Gate)**：每一個工具呼叫在執行前，都必須通過權限系統的 7 層檢查。
8.  **執行與結果收集 (Execution & Result Collection)**：工具實際與本機環境互動（如跑測試、改程式碼），並將結果寫回 `tool_result`。
9.  **終止條件判定 (Stop Condition)**：如果模型回傳純文字且不再呼叫工具，或者觸發了最大回合限制（Max turns），迴圈終止並回覆使用者。

**💡 流程圖：Agentic Loop 的心跳跳動**
```text
[ START: User Prompt ]
       │
       ▼
 ┌───[ Context ] <─── (讀取設定與歷史)
 │     │
 │  [ Shapers ] ───── (五層壓縮管線預先節流)
 │     │
 │  [ Model ] ─────── (Claude API 串流回傳)
 │     │
 │  [ Decision ] ──── < 有工具呼叫嗎？ > ─── (NO) ──> [ END: 回覆人類 ]
 │     │                                         
 │    (YES)                                      
 │     │                                         
 │  [ Safety ] ────── (權限檢查與沙盒過濾)
 │     │                                         
 │  [ Action ] ────── (執行 Bash / LSP / 讀寫檔案)
 │     │                                         
 └── [ Result ] ────── (將結果寫入歷史，回到 Context 步驟)
```

### 2.2 並發執行與效能管理 (StreamingToolExecutor)

在真實的開發情境中，模型可能會一次提出多個工具請求（例如：同時讀取 3 個不同的檔案）。為了提高效率而不犧牲系統的穩定性，Harness 實作了 `StreamingToolExecutor` (Sec 4.2)。

**核心機制：並發讀取，序列寫入 (Parallel Read / Serial Write)**
Harness 會將工具標記為兩種性質：
*   **並發安全 (Concurrent-safe)**：主要是唯讀操作（如 `FileRead`）。Harness 會同時啟動多個執行緒平行處理，大幅降低 I/O 等待時間。
*   **排他執行 (Exclusive)**：會改變狀態的操作（如執行 `BashTool` 的寫入指令或修改檔案）。Harness 會確保這些動作按照順序「一個一個排隊執行」，防止競態條件（Race conditions）導致程式碼寫爛。

**關鍵工程設計：手足中止控制器 (Sibling Abort Controller)**
這是一個極具實戰意義的設計。如果模型同時發出了 3 個 Bash 指令，當其中一個指令報錯（Error）時，Harness 會自動觸發中止訊號，**立刻切斷剩下 2 個還在跑的子程序**。
*   **優點**：避免無意義的運算浪費，並防止錯誤擴散。

### 2.3 錯誤恢復與系統韌性 (Graceful Recovery)

Agent 在執行過程中一定會遇到極端情況（Edge Cases）。一個好的 Harness 不能只是崩潰並噴出 Stack Trace，它必須具備「自癒能力」。

根據論文 **Sec 4.4**，Claude Code 內建了以下三種優雅恢復機制：

1.  **輸出 Token 擴展 (Max Output Tokens Escalation)**：
    *   **場景**：模型回傳程式碼到一半，因為達到 `max_tokens` 限制而卡住。
    *   **恢復**：Harness 會捕捉到這個中斷，自動調高輸出的 Token 上限，並在同一個回合內重試，最多嘗試 3 次。
2.  **反應式壓縮 (Reactive Compaction)**：
    *   **場景**：當 API 返回 `prompt_too_long`（提示詞過長）錯誤時。
    *   **恢復**：Harness 會立即跳過輕量壓縮，直接執行最強力的壓縮（如 L4 或 L5），將 Context 瘦身後自動重新發送請求。這就是我們之前討論過的「失敗後的強制重試」。
3.  **降級模型 (Fallback Model)**：
    *   **場景**：當最強的主模型（如 Sonnet）因為網路或服務異常失效時。
    *   **恢復**：Harness 會自動降級使用較小的模型（如 Haiku）來維持系統的運作。

---

**👨‍🔬 本章學習要點：**
*   Agent 的迴圈是「狀態感知」的。
*   利用 `StreamingToolExecutor` 在效率（平行讀取）與安全（序列寫入）之間取得平衡。
*   Harness 的價值在於處理 API 報錯、Token 溢出與並發管理，讓模型能心無旁鶩地進行推理。

## 3. 安全防禦與權限邊界 (Safety & Permission Gate)

AI Agent 最強大但也最危險的地方在於它能執行 Shell 指令與讀寫檔案。Anthropic 的內部數據指出一個殘酷的現實：**使用者核准權限請求的機率高達 93% (Sec 5.1)**。這種「授權疲勞 (Approval Fatigue)」意味著人類不再是可靠的守門員。因此，Claude Code 的架構設計核心是：**「權限控制必須由 Harness 獨立執行，不論使用者是否在看。」**

### 3.1 預設拒絕 (Deny-First) 與七層安全防護漏斗

Claude Code 採用 **「深度防禦 (Defense in depth)」** 原則。模型發出的每一個 `tool_use` 請求都必須穿越一個由 7 個獨立階段構成的過濾漏斗：

```text
[ 模型發出請求：執行某個工具 ]
       │
       ▼
 1. 【工具預先過濾 (Pre-filtering)】 ─────── (在發送 Prompt 前就將禁用工具剔除)
       │
       ▼
 2. 【拒絕規則引擎 (Deny-first Rules)】 ───── (Deny 規則永遠優先於 Allow)
       │
       ▼
 3. 【Hook 攔截器 (Hook Interception)】 ───── (PreToolUse Hook 程式化審核)
       │
       ▼
 4. 【權限模式約束 (Permission Modes)】 ───── (根據目前的信任模式決定是否彈窗)
       │
       ▼
 5. 【ML 分類器 (ML Classifier / Auto-mode)】─ (獨立的小模型進行意圖安全性分析)
       │
       ▼
 6. 【作業系統沙盒 (Shell Sandboxing)】 ───── (最後的物理隔離，限制檔案與網路存取)
       │
       ▼
 7. 【會話邊界遺忘 (Session Reset)】 ──────── (跨對話重啟後，所有已授權權限歸零)
       │
       ▼
[ 🟢 成功：安全執行 ]
```

*[📝 圖片提示：請在此處插入論文中的 Figure 4 Permission gate overview，展示規則與 Hook 如何介入決策]*

### 3.2 漸進式信任光譜：7 種權限模式

為了在「安全性」與「開發效率」之間取得平衡，Claude Code 實作了一套漸進式的信任光譜 (Sec 5.1)。這 7 種模式決定了系統對 AI 的放任程度：

| 模式名稱 | 自主性 | 核心行為描述 |
| :--- | :---: | :--- |
| **`plan`** | 極低 | AI 僅能提案，所有行動（即使是讀檔）都必須人類手動確認。 |
| **`default`** | 低 | 預設模式。大部分具有副作用的操作都需要彈窗確認。 |
| **`acceptEdits`** | 中 | **條件式自動**。允許直接修改檔案，但執行 Shell 指令仍需確認。 |
| **`auto`** | 中高 | 啟動 **ML 分類器**。由小模型評估安全性，若判斷無害則自動放行。 |
| **`dontAsk`** | 高 | 不再跳出詢問視窗，但 `Deny` 寫死的規則依然會強制執行。 |
| **`bypassPermissions`** | 極高 | 跳過幾乎所有權限檢查，僅保留核心安全性硬限制。 |
| *(內部) `bubble`* | 特殊 | **子代理專用**。將權限請求「向上冒泡」回傳給母會話的人類決定。 |

**💡 關鍵觀念：Reversibility-weighted (基於可逆性的權衡)**
Harness 在設計這些模式時考慮了動作的「可逆性」。修改程式碼（Git 可以還原）是高可逆的，所以 `acceptEdits` 會放行；但跑 `npm install` 可能下載病毒（不可逆），所以必須更嚴格。

### 3.3 拒絕即路由 (Denial as a Routing Signal)

這是一個極具智慧的架構設計。在 Claude Code 中，權限被拒絕不代表系統崩潰或報錯，而是一個 **「導航訊號」**。

**💡 舉例說明：**
1. **模型意圖**：試圖執行 `rm -rf /` 來清理空間。
2. **Harness 反應**：`Deny-first` 引擎偵測到危險指令，觸發 `PermissionDenied` 事件。
3. **注入反思**：Harness 不報錯，而是回傳一段 `tool_result` 給模型：「*這項操作具有危險性，已被安全策略擋下。請改用更安全的方法或解釋你的意圖。*」
4. **模型修正**：模型收到結果後進行「自我反思」，決定改用 `FileRead` 逐一檢查檔案再刪除。

**結論**：Harness 的角色不是「警察」，而是帶有邊界的「教練」，透過拒絕引導模型採取更專業、更安全的工程路徑。

### 3.4 深度防禦的隱患：時序漏洞與效能衝突

作為技術文件，我們必須指出目前架構的三個已知弱點，這是未來 Agent 設計者的重要教訓：

1. **效能導致的安全降級 (Performance vs. Safety)**：
   * **問題**：資安研究指出，若一個 Shell 指令包含超過 50 個子指令，Harness 為了防止 UI 卡死（Parsing overhead），會**自動跳過詳細規則比對**，退回到單一的通用詢問。
   * **學習點**：當防禦機制的效能成為瓶頸時，安全性會發生「級聯崩潰」。
2. **初始化時序漏洞 (The Pre-trust Gap)**：
   * **問題**：系統在「專案初始化（載入 Plugin/Hook）」階段，為了效率，是在「權限對話框」出現之前就執行的。
   * **學習點**：這導致了一個**時間窗漏洞**。駭客可以將惡意代碼植入 Hook 的初始化函式中，在保護機制啟動前就完成攻擊。
3. **監督悖論與授權疲勞**：
   * **問題**：使用者對系統的信任會隨時間「 habituation（習慣化）」。當使用者習慣了按 `Y`，防護網的前 4 層其實已經形同虛設，系統的最後防線被迫退縮到 **ML 分類器** 與 **沙盒隔離** 上。

---

**👨‍🔬 本章學習要點：**
*   **不信任原則**：Harness 必須獨立於使用者的警覺性。
*   **多層過濾**：利用 7 層漏斗將「人類疲勞」的風險降至最低。
*   **路由邏輯**：權限拒絕應作為提示詞的一部分，引導 Agent 自我修正。
*   **時序重要性**：初始化階段的權限管理是目前 Agent 的最大攻防前線。

## 4. 大腦容量管理：上下文與記憶體管線 (Context & Memory Management)

在 Agent 系統中，**「上下文視窗 (Context Window)」是最稀缺且昂貴的資源**。Claude Code 雖然支援高達 200K 甚至 1M 的長度，但頻繁的讀檔與執行報錯會瞬間填滿容量。為此，Harness 實作了一套類似作業系統記憶體管理的「層次化加載」與「漸進式回收」機制。

### 4.1 上下文視窗的生命週期與動態組裝

Harness 在呼叫模型前，會根據不同的 **「刷新生命週期」** 將資訊拼裝進 Context 中。這不是每次都重新讀取，而是透過快取 (Memoization) 來極大化效能。

**💡 核心架構：Context 的六大組成與刷新頻率**

| 組成部分 | 刷新頻率 | 包含內容 |
| :--- | :--- | :--- |
| **1. 系統提示詞** | **會話快取 (Once)** | 核心規則、API 指令集。 |
| **2. 專案設定** | **會話快取 (Once)** | 四層 `CLAUDE.md` 規則、Git 基本狀態。 |
| **3. 自動記憶** | **每回合動態注入** | 透過小模型預取的相關專案知識檔案。 |
| **4. 對話歷史** | **持續累積** | 之前的 User Prompt 與 Assistant 回覆。 |
| **5. 執行結果** | **持續累積** | 讀取的程式碼、Bash 輸出、子代理總結。 |
| **6. 延遲工具** | **按需觸發 (On-demand)** | 完整工具 Schema (僅在呼叫 `ToolSearch` 後)。 |

*[📝 圖片提示：請在此處插入論文中的 Figure 6 Context construction and memory hierarchy，展示不同 mutability 的資源如何匯聚到模型大腦]*

### 4.2 透明的靜態記憶：`CLAUDE.md` 階層機制 (Sec 7.2)

Claude Code 拒絕使用不透明的向量資料庫 (Vector DB) 作為主要記憶，而是採用 **「純文字 Markdown 檔案」**。這確保了開發者對 AI 的記憶具有 **100% 的審計權與修改權**。

**四層優先級加載 (Reverse order of priority)**：
系統會由遠到近掃描 `CLAUDE.md`，**越靠近工作目錄的檔案越晚載入**。這是利用了 LLM 的 **「近因偏差 (Recency Bias)」**——越晚出現的資訊，模型注意力越高，權重越高：
1. `Managed` (/etc/...)：系統級規範。
2. `User` (~/.claude/...)：個人習慣。
3. `Project` (./CLAUDE.md)：專案共用規則。
4. `Local` (./CLAUDE.local.md)：私有敏感設定。

### 4.3 零依賴的動態記憶：自動記憶 (Auto Memory) 與延遲加載

為了在不塞爆 Context 的前提下讓 AI 變聰明，系統實作了兩套動態注入機制：

1.  **自動記憶 (Auto Memory)**：
    *   **機制**：Harness 會在背景發送一個非同步請求，讓一個 **小型 LLM** 快速掃描所有記憶檔案的「標題」。
    *   **過程**：模型根據使用者的 Prompt 挑選出最相關的 1~5 份文件，並在「封箱發車前最後一秒」將內容注入 Context (Late Injection)。
2.  **工具說明書延遲加載 (Deferred Schemas)**：
    *   **機制**：一開始只給模型看「工具名單」。
    *   **過程**：只有當模型主動呼叫 `ToolSearch` 以了解某個 MCP 工具的細節時，Harness 才會載入幾千字的完整說明書。這避免了在 Context 中裝載「模型沒打算用的工具」。

### 4.4 核心魔法：五層漸進式壓縮管線 (Compaction Pipeline)

當 Token 壓力接近極限時，Harness 會啟動 **「懶惰降級 (Lazy-degradation)」** 管線，按照 1 到 5 的順序，從最便宜到最昂貴、從物理性到語義性地壓榨空間。

**📊 五層管線執行矩陣**

| 級別 | 名稱 | 實作細節 | 具體舉例 |
| :--- | :--- | :--- | :--- |
| **L1** | **預算縮減** | 截斷單一過長的輸出。 | 把萬行報錯 Log 縮減成首尾各 500 行。 |
| **L2** | **裁切 (Snip)** | 暴力移除最舊的對話。 | 直接丟棄對話剛開始的前 10 個回合。 |
| **L3** | **微壓縮** | 結構化 Metadata 微縮。 | **快取感知優化**：縮減 ID 且不破壞 Prefix。 |
| **L4** | **上下文折疊** | 建立「虛擬視圖」。 | 用「摘要」暫時替換「原文」，原文仍在磁碟。 |
| **L5** | **自動總結** | **呼叫模型重寫摘要**。 | 徹底燒掉舊記憶，換成一段 500 字的精華。 |

### 4.5 壓縮與快取的極致平衡：Microcompact 與快取驅逐 (Cache Eviction)

這是系統工程中最精妙的地方：**如何在壓縮內容時，不破壞 LLM 的 Prompt Caching？**

**💡 解決方案：先發車、後補票 (Deferred Boundary)**
1.  **問題**：只要壓縮改了一個字，Prefix 就會改變，導致 Cache Miss，系統變慢且變貴。
2.  **L3 Microcompact 策略**：它在「回合 11」發現壓力時，忍住不改，原封不動發送。
3.  **命中快取**：回合 11 成功白嫖了一次 Cache Hit。
4.  **精準校時**：等 API 回傳後，Harness 讀取 Headers 裡的 `cache_deleted_input_tokens` (收據)，確認伺服器精確丟棄了多少 Token。
5.  **更新狀態**：此時才在本地建立 `compact_boundary` 與 **UUID 鏈路**，確保「回合 12」能以最精準的帳本重新對齊快取。

> ⚠️ **失敗恢復機制**：
> 若在「白嫖」過程中不幸真的撞牆（伺服器噴回 `prompt_too_long`），Harness 會觸發 **「反應式重試 (Reactive Retry)」**，立刻執行 L5 強制壓縮並自動重新發送。

*   **Context 是 RAM**：需要層次化載入與 Lazy-loading 策略。
*   **不信任向量資料庫**：Markdown 檔案是人類維護 Agent 記憶的最佳媒介。
*   **五層管線的智慧**：從物理裁切到語義總結，每一層都在權衡成本與細節遺失。
*   **快取至上**：L3 的延遲記帳技術是生產級 Agent 穩定性與效能的護城河。

## 5. 肢體擴展與多智能體協作 (Extensibility & Subagents)

一個生產級的 Agent 不能只會寫程式，它必須能與外部世界（如資料庫、Slack、搜尋引擎）溝通。Claude Code 的核心設計思想是：**「不同的功能擴展，應根據其對 Context 的負擔，採取不同的接入機制。」**

### 5.1 漸進式上下文成本的四種擴展機制 (Sec 6)

Harness 將擴展機制分為四類，這四類分別插在 `queryLoop` 的不同生命週期中。

**📊 擴展機制深度對比表 (Ref: Table 2)**

| 機制名稱 | 核心能力 (Capability) | **Context 成本** | 插入點 (Insertion Point) |
| :--- | :--- | :---: | :--- |
| **Hooks** | 生命週期攔截、事件自動化。 | **零 (Zero)** | `execute()`: 工具執行前後。 |
| **Skills** | 特定領域的指令集與複合動作。 | **低 (Low)** | `assemble()`: 上下文注入。 |
| **Plugins** | 多組件打包與分發的容器。 | 中 (Medium) | 橫跨迴圈所有點位。 |
| **MCP** | 外部服務整合 (跨語言、跨網路)。 | **高 (High)** | `model()`: 工具池組裝。 |

*[📝 圖片提示：請在此處插入論文中的 Figure 5，展示這些機制如何插進 a. assemble()、b. model()、c. execute() 三大注入點]*

### 5.2 深入細節：從神經反射到外部義肢

1.  **Hooks (鉤子) —— 系統的「神經反射」**：
    *   **原理**：Hook 是寫死在 Harness 裡的程式碼。當特定事件觸發（如 `PreToolUse`），Harness 直接執行邏輯。
    *   **優勢**：因為是程式碼邏輯，模型完全感覺不到，**Token 成本為零**。
    *   **實戰**：`PreToolUse` Hook 可以偷偷改寫模型想執行的危險指令，或是用「小模型」做安全審查（`prompt` 類型命令）。
2.  **Skills (技能) —— 模型的「專業手冊」**：
    *   **原理**：每個 Skill 由 `.md` 檔案定義。模型呼叫 `SkillTool` 時，Harness 才載入手冊全文。
    *   **機制**：採用 **「延遲技能發現」**。初始只載入「名稱+一句話描述」，模型點菜後才載入數千字的全武功秘笈。
3.  **MCP (模型上下文協定) —— 系統的「USB 接口」**：
    *   **原理**：標準的 JSON-RPC 協定，讓 Claude 連結 Google Search 或 Postgres。
    *   **機制**：為了省錢，MCP 同樣支援 **「延遲工具定義」**。只有模型想用時，才透過 `ToolSearch` 要說明書。

### 5.3 核心組件：Plugin (外掛套件) 作為功能容器

在原始碼中，**Plugin 不是一個功能，而是一個包裝盒**。一個 `PluginManifestSchema` 可以同時打包 10 種組件：

*   **斜槓指令** (如 `/stats`)
*   **自定義代理人** (Agents)
*   **技能組** (Skills)
*   **事件監聽器** (Hooks)
*   **外掛伺服器** (MCP/LSP)
*   **UI 樣式與輸出頻道** (Output Styles/Channels)

> 💡 **學習點**：Plugin 機制讓第三方開發者能一次性賦予 Agent「新的眼睛（LSP）」、「新的手（MCP）」與「新的反射（Hooks）」。

### 5.4 專業大腦：LSPTool 的深度整合

為什麼 Claude Code 比一般 Chatbot 更懂程式碼？因為它內建了 **LSP (Language Server Protocol)**。
*   **精準導航**：它不靠 Grep 瞎找，而是呼叫 `go_to_definition` 精準跳到函數定義。
*   **自我修復**：AI 改完 code 後，立刻呼叫 `get_diagnostics` 詢問編譯器有沒有錯。
*   **節省 Token**：它不需要讀取整個檔案，只要呼叫 `hover` 拿取該函數的型別定義（Signature）即可。

### 5.5 子代理 (Subagent) —— 召喚分身與隔離架構 (Sec 8)

當任務涉及多個檔案或高風險實驗時，母代理會呼叫 `AgentTool` 生出子代理。這是一套極度嚴謹的協作架構：

**1. 物理級隔離 (Isolation Sandbox)**
*   **Git Worktree 模式**：子代理在一個獨立的 Git 分支/資料夾工作。不管它把程式碼改得多爛、測試跑得多崩潰，**人類在主視窗看到的原始碼都是安全的**。

**2. 權限向上冒泡 (Permission Bubbling)**
*   如果子代理想跑一個危險指令，它會觸發 `bubble` 模式。
*   這個權限請求會傳回母會話，最後由人類點擊確認。這確保了 **「權限控制始終由人類終端掌握」**。

**3. 僅限總結回傳 (Summary-only Return)**
*   子代理在大腦裡打了一場 5 萬 Token 的大戰。
*   **重點**：結束後，子代理**燒掉所有歷史**，只回傳 300 字的報告。
*   **數據**：這避免了子代理的「噪音」把母代理的 Context 撐爆（子代理消耗量通常是單人的 7 倍）。

**💡 流程圖：子代理協作流**
```text
[ 母代理：遇到大任務 ] 
       │
       ▼
[ 呼叫 AgentTool ] ───> [ 建立獨立 Context ]
       │                [ 建立 Git Worktree ]
       │
       ▼
[ 子代理執行任務 ] <─── [ 權限不足時向母代理冒泡 ]
       │
       ▼
[ 任務結束 ] ────┐
       │         │ (燒掉數萬字對話歷史)
       ▼         ▼
[ 回傳 Summary ] ───> [ 寫入母代理 Context ]
```

---

**👨‍🔬 本章學習要點：**
*   **按需載入 (On-demand)**：無論是 Skill 還是 MCP，沒用到前絕不塞進大腦。
*   **Plugin 是包裝盒**：它是組件的分發中心。
*   **LSP 是工程師的第三隻眼**：它提供絕對精準的程式碼語意。
*   **子代理是「發包中心」**：透過 Git Worktree 物理隔離與總結回傳，在安全與 Token 之間取得平衡。

## 6. 狀態保存與會話持久化 (State Persistence & Recovery)

在一個生產級的 Agent 系統中，持久化（Persistence）不僅是為了「存檔」，更是為了「狀態重建」。Claude Code 捨棄了傳統的關聯式資料庫，轉而採用一種極度透明、高審計性的 **「僅限附加 (Append-only)」** 日誌哲學。這確保了即便在系統崩潰或被壓縮後，人類依然能 100% 還原 AI 的思考路徑。

### 6.1 僅限附加的日誌哲學 (Append-only JSONL)

Claude Code 的所有數據都存儲在 `.jsonl` (JSON Lines) 格式的檔案中。這是一個極具工程智慧的選擇：

*   **高審計性 (Auditability)**：每一行都是一個獨立的 JSON 物件。開發者可以隨時用 `tail`、`grep` 或任何文字編輯器打開對話紀錄檔，完全不依賴專門的資料庫檢視工具。
*   **Git 友好**：每一行代表一個事件，這讓對話紀錄非常容易被納入版本控制，差異對比（Diff）非常清晰。
*   **三條獨立的數據通道**：
    1.  **會話劇本 (`[sessionId].jsonl`)**：存放在專案目錄，包含該 Session 的所有細節。
    2.  **全域提示詞歷史 (`history.jsonl`)**：存放在使用者家目錄，僅存 Prompt，用於 CLI 的 `Ctrl+R` 搜尋。
    3.  **子代理側鏈 (`[subagentId].jsonl`)**：獨立存放子代理的「碎碎念」，避免污染主對話紀錄。

### 6.2 核心重點：為何必須存儲「工具執行結果」？

這是一個常被初級工程師忽略的細節：**對話紀錄檔裡不能只存對話，必須存儲「模型看到的一切」。**

在 `[sessionId].jsonl` 中，Harness 會完整記錄每一筆 `tool_result`。
*   **場景還原**：假設 AI 曾讀取 `config.ts`（500 行），並根據內容改了 code。
*   **重建必要性**：當你隔天 `--resume`（恢復對話）時，模型的大腦是空的。如果 JSONL 裡沒有存那 500 行程式碼，模型就不知道它昨天為什麼要那樣改，會導致嚴重的幻覺（Hallucination）。
*   **預算縮減掛鉤**：如果工具回傳結果（如 Log）太長，JSONL 會存儲一個 **「內容參照標籤」**。系統在恢復時會依據標籤去尋找原始數據。

**💡 JSONL 紀錄示例：**
```json
{"type": "user", "content": "幫我修復 bug", "ts": 1715000000}
{"type": "tool_use", "id": "call_01", "name": "FileRead", "params": {"path": "A.ts"}}
{"type": "tool_result", "tool_use_id": "call_01", "content": "export const a = 1; ... (500行)", "ts": 1715000010}
{"type": "assistant", "content": "我看到了 A.ts，我準備修改它...", "ts": 1715000020}
```

### 6.3 壓縮邊界的 UUID 鏈路修補 (Read-time Chain Patching)

我們在 Phase 4 提到過 L5 級別的壓縮會「燒掉」舊歷史換成摘要。那麼，恢復會話時，系統怎麼知道哪些是被燒掉的？哪些是剩下來的？

Harness 在 JSONL 裡插入 `compact_boundary` 事件，並攜帶三種關鍵 UUID：
*   **`headUuid`**：指向被壓縮段落的開頭。
*   **`anchorUuid`**：將目前這段「摘要」錨定在正確的時間點。
*   **`tailUuid`**：指向壓縮段落的結尾。

**運作流程：**
當 `conversationRecovery.ts` 載入檔案時，它會像玩拼圖一樣：
1. 讀到摘要物件。
2. 看到 UUID 指標。
3. 自動跳過硬碟中那些已被標記為「已壓縮」的原始訊息。
4. 在記憶體中重新拼接（Patch）訊息的 `parentUuids` 關聯。
5. **最終結果**：雖然硬碟檔案是碎裂的，但送進大模型的是一條邏輯完美連貫的記憶長鏈。

### 6.4 安全至上的不變性：為什麼不恢復權限？

這是 Claude Code 在架構設計上的一個「反直覺」但「高安全性」的選擇。

*   **現象**：當你使用 `--resume` 或 `fork` 恢復會話時，AI 會記得所有的聊天內容，但 **Harness 會遺忘所有你按過的 Allow/Deny 授權**。
*   **工程原因**：這被稱為 **「安全不變性 (Safety Invariant)」**。
    *   系統認為：**「會話（Session）是獨立的信任網域」**。
    *   昨天的環境是安全的，不代表今天的環境也是安全的（例如專案可能在半夜被植入了惡意指令）。
    *   **代價**：使用者必須對同一個危險動作重新點擊一次同意。
    *   **收益**：確保每一項敏感操作的授權都是基於「當下環境」的清醒決策，防止過期權限被惡意利用。

### 6.5 分支、回溯與文件快照 (Fork & Rewind)

除了對話紀錄，Claude Code 還提供了對檔案系統的保護：

| 功能 | 機制 | 作用 |
| :--- | :--- | :--- |
| **`fork`** | 複製 JSONL | 建立一個平行的對話空間，實驗失敗後可隨時切回原本的分支。 |
| **`rewind-files`** | `checkpoints` | 在 `~/.claude/file-history/` 下存放檔案快照，可將程式碼還原到特定對話點。 |

**💡 流程圖：狀態恢復的全景圖**
```text
[ 啟動 --resume ]
       │
       ▼
[ 1. 讀取 sessionId.jsonl ] ───> [ 2. 透過 UUID 鏈路修補，過濾掉已壓縮訊息 ]
       │                                     │
       ▼                                     ▼
[ 4. 重新組裝對話與工具結果 ] <─── [ 3. 獲取最終摘要 (Final Summary) ]
       │
       ▼
[ 5. 重建 Model Context ] ────> [ 6. ⚠️ 權限歸零，等待使用者下一個指令 ]
```

---

**👨‍🔬 本章學習要點：**
*   **JSONL 是萬能的**：它兼顧了透明度、Git 友好度與寫入效能。
*   **狀態重建 (Reconstruction)**：持久化必須包含「模型當時看到的環境數據（工具結果）」。
*   **UUID 鏈路**：這是解決「壓縮後如何重讀歷史」的進階資料結構技巧。
*   **安全性原則**：為了防護，寧願犧牲便利性也不要在恢復會話時繼承權限。

## 7. 宏觀架構對比：Task-level vs. Gateway-level

一個系統的架構長相，並非由技術流行趨勢決定，而是由其 **「信任模型 (Trust Model)」** 與 **「生命週期」** 決定的。透過 Claude Code 與 OpenClaw 的對比，我們能看見 Agent 設計空間 (Design Space) 的兩極。

### 7.1 單一環境 (Claude Code) vs. 多渠道網關 (OpenClaw)

這兩者的差異，可以用 **「單幹的私家偵探」** 與 **「大型跨國飯店總機」** 來比喻。

**📊 六大架構維度碰撞表 (Ref: Table 3)**

| 維度 | **Claude Code (偵探型)** | **OpenClaw (飯店型)** |
| :--- | :--- | :--- |
| **系統角色** | 專注於單一專案的 **Coding Harness**。 | 多渠道的 **個人助理網關 (Gateway)**。 |
| **系統生命週期** | **短暫 (Ephemeral)**：跟隨 CLI 生滅。 | **常駐 (Persistent)**：身為 Daemon 運作。 |
| **信任模型** | **不信任模型**：假設 AI 會做壞事。 | **不信任外部**：假設外來訊息有詐。 |
| **安全邊界** | **動作級別 (Per-action)**：7 層漏斗。 | **周邊級別 (Perimeter)**：白名單、配對。 |
| **擴展對象** | 擴充 **單一 Agent** 的上下文與工具。 | 擴充 **整個網關** 的能力（所有 Agent 共享）。 |
| **記憶管理** | 專案 Markdown + 5 層壓縮管線。 | 結構化長期記憶 + **Dreaming (做夢機制)**。 |

### 7.2 權力中心的轉移：Loop-centric vs. RPC-centric

這是兩者在原始碼結構上最大的不同：

1.  **Claude Code：以「迴圈」為中心**
    *   `queryLoop` 是宇宙的中心。所有的 UI 顯示、工具執行、權限檢查，全部都是為了服務這個迴圈。這適合處理需要高度專注、長路徑推理的「重度任務」。
2.  **OpenClaw：以「RPC 分發層」為中心**
    *   網關 (Control Plane) 才是老大。Agent 迴圈在裡面只是一個被叫用的「插件小弟 (Pi-agent runner)」。
    *   **為什麼？** 因為網關要同時處理來自 WhatsApp、Telegram、Slack 的訊息。它必須先做排隊 (Queue)、身分識別、Session 解析，確認沒問題後，才把工作發包給內部的 Agent 迴圈處理。

### 7.3 記憶與擴展的哲學分歧

*   **記憶機制**：
    *   Claude Code 為了省錢，發展出極致的 **「上下文壓縮 (Compaction)」** 技術。
    *   OpenClaw 為了生活助理的長期需求，發展出 **「記憶晉升 (Promotion)」** 技術。它會利用 LLM 在背景「做夢」，把昨天的瑣事提煉成永恆的知識（存入 `MEMORY.md`）。
*   **能力擴展**：
    *   Claude Code 的外掛（如 MCP）是為了讓「這個 AI」能修 Bug。
    *   OpenClaw 的外掛（如圖像生成、語音轉文字）是一次註冊、全家享用。這是一種 **「網關級能力註冊」** 的思路。

### 7.4 架構的層次化組合：俄羅斯娃娃 (The Russian Doll)

論文在 Sec 10.2 提出了一個最深刻的洞見：**這兩套架構並非競爭關係，而是可以互相「嵌套」的。**

OpenClaw 支援一種協定叫 **ACP (Agent Client Protocol)**。這意味著：
*   你可以讓 OpenClaw 擔任你的「前端管家」。
*   當你透過手機 WhatsApp 下令：「幫我修復專案 bug」時。
*   OpenClaw 網關會啟動一個 **Claude Code 執行緒** 作為它的子系統（Harness）。

**💡 啟發：** 未來的 AI Agent 生態不會只有一種架構，而是由 **「網關級系統 (管理溝通與記憶)」** 包裹著 **「任務級框架 (管理執行與安全)」**。

## 8. 商業級防禦攻防與工程師文化 (Commercial Defense & Culture)

如果說前六章是在教你如何蓋一棟堅固且智能的摩天大樓，第八章就是在教你如何防止商業間諜潛入，以及這群蓋大樓的工程師（包含 AI）有著什麼樣的獨特文化。這部分資訊主要源自於對 Claude Code 原始碼中隱藏標籤與底層二進位實作的逆向分析。

### 8.1 主動毒害抄襲者：反蒸餾機制 (Anti-Distillation)

在 AI 領域，最大的威脅不是程式碼被偷，而是「模型智慧被蒸餾（Distillation）」。競爭對手可以用便宜的模型大量抓取 Claude Code 的輸入與輸出，來訓練他們自己的模型。

**💡 核心策略：在「湯」裡下毒 (Data Poisoning)**
為了對抗這種行為，Claude Code 在發送 API 請求時實作了兩套防禦：

1.  **注入「假工具 (Decoy/Fake Tools)」**：
    *   Harness 會在 System Prompt 的工具清單中，隨機塞入幾個**「根本不存在」**的虛擬工具定義。
    *   **效果**：如果對手把這段資料拿去訓練，他們的模型就會學到錯誤的知識，並在未來試圖呼叫這些不存在的工具，導致對手的系統崩潰或產生嚴重的幻覺。
2.  **隱藏推理過程 (CONNECTOR_TEXT)**：
    *   AI 最值錢的 `<thinking>` 過程在傳輸給客戶端前，會在伺服器端被過濾或加密成 `CONNECTOR_TEXT`。這確保了即便客戶端程式碼被破解，也拿不到那套最寶貴的「解題思路」。

### 8.2 超越 JS 的底層授權保護：DRM in Zig

Claude Code 雖然是用 TypeScript 寫的，但 TS 像「玻璃屋」一樣透明，極易被破解（Monkey-patching）。Anthropic 採用了**「跨語言授權驗證」**來解決這個問題。

**📊 防禦層級對比：玻璃屋 vs. 保險箱**

| 層級 | 實作語言 | 安全性 | 作用 |
| :--- | :--- | :--- | :--- |
| **邏輯層** | **TypeScript** | **低** | 容易被修改 `if(license) => if(true)`。 |
| **執行層** | **Bun (Zig 編寫)** | **極高** | 編譯成二進位機器碼，逆向工程極難。 |

**💡 關鍵實作細節：偷天換日 (The Hijacking)**
*   **誘餌標籤**：TS 原始碼中會發送一個假字串（如 `cch=ed1b0`）。駭客以為只要模擬這個字串就能白嫖。
*   **底層重寫**：當請求準備離開電腦前，底層用 **Zig 語言** 寫成的執行引擎（Bun）會突然介入，利用預先編譯好的算法，計算出真正的加密雜湊值（Cryptographic Hash）並**直接覆寫**掉那個誘餌標籤。
*   **結論**：要破解這套認證，駭客必須撬開 Zig 編譯出來的「鈦合金保險箱」，這將攻擊成本提升了數個量級。

### 8.3 系統邊界防護：動靜態 Prompt 邊界與臥底模式

1.  **快取邊界管理 (`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`)**：
    *   Harness 在代碼中明確定義了哪些部分是「可快取的靜態規則」，哪些是「危險的無快取動態資訊」（如當下時間、Git 分支）。
    *   這確保了開發團隊在更新 Prompt 時，不會不小心把頻繁變動的變數塞進靜態區，導致公司 API 費用飆升。
2.  **臥底模式 (Undercover Mode)**：
    *   系統內建 `undercover.ts`。當 Anthropic 員工用 Claude Code 參與外部開源專案時，此模式會強制啟動且**不可關閉**。
    *   它會自動抹除所有內部模型代號（如 `Capybara`），且絕不承認內容是由 AI 生成。這既是商業保護，也是一種特殊的網路倫理實踐。

### 8.4 魔法文檔 (Magic Docs) 與 LLM 導向註解

這部分展示了「當工程師擁有了頂級 Agent」後，開發文化會如何演變：

*   **魔法文檔 (`MAGIC DOC` 標籤)**：
    *   工程師在 Markdown 文件頭加上這個標籤。當系統閒置時，會自動喚醒一個**背景子代理人**，根據最近的代碼變更自動重寫並更新這份文件。
    *   **學習點**：實現了「文檔與代碼同步」的工程終極理想。
*   **寫給 AI 看的註解 (LLM-oriented Comments)**：
    *   原始碼中出現大量 `// NOTE TO LLM` 的註解。這說明程式碼的讀者不再只是人類，開發者開始有意識地引導 AI 該如何理解這段複雜的異步邏輯。

**💡 流程圖：商業防禦與自動化文化的交織**
```text
[ 開發者修改代碼 ]
       │
       ▼
 ┌───[ 觸發背景任務 ] ───> [ 檢查 MAGIC DOC ] ───> [ AI 自動更新規格書 ]
 │
 ▼
[ 發送 API 請求 ]
       │
       ▼
 ┌───[ TypeScript ] ───> [ 注入假工具 (防蒸餾) ]
 │     │
 └───[ Zig (Bun) ] ────> [ 計算加密雜湊 (防破解) ] ───> [ 出海送往 Anthropic ]
```

*   **主動防禦**：在 Prompt 裡「下毒（假工具）」比單純隱藏原始碼更有效。
*   **跨語言校驗**：利用低階編譯語言（Zig）來守護高階直譯語言（TS）的商務邏輯。
*   **AI 本位開發**：Agent 不只是工具，它已經深入到文件維護與註解編寫的文化核心。

## 9. 系統反思與未來演進方向 (System Reflections & Future)

透過對 Claude Code 的深度解剖，我們看見了一個極度成熟的「反應式 (Reactive)」Agent 架構。然而，技術的演進永不停止。本章將探討 Agent 如何從「被動工具」轉化為「主動員工」，並反思目前架構中尚未解決的終極挑戰。

### 9.1 走向主動型代理：KAIROS 系統與心跳機制

目前大多數 Agent（包含目前的 Claude Code 預設模式）都是 **「不點不亮」** 的：使用者輸入一個指令，AI 動作一次。但原始碼中隱藏的 **KAIROS** 架構展示了下一步。

**💡 核心進化：從 Prompt 驅動到 Tick 驅動**
*   **心跳 (Tick-based Heartbeats)**：當使用者不在電腦前時，Harness 會定時向模型注入一個 `<tick>` 訊號。
*   **自主判斷**：模型收到 `<tick>` 後，會檢查當前專案是否有掛起的任務（例如正在跑的編譯、待更新的文件）。
*   **經濟節流閥 (`SleepTool`)**：主動型 Agent 面臨的最大問題是「錢（Token）會燒完」。KAIROS 引入了 `SleepTool`：
    *   如果模型判斷暫時沒事做，它會呼叫 `SleepTool`。
    *   這會讓系統進入休眠，暫停 `<tick>`，直到快取快過期或使用者回來為止。

**📊 Agent 演進對比表**

| 特性 | **第一代 (Chatbot)** | **第二代 (Reactive Agent)** | **第三代 (Proactive Agent)** |
| :--- | :--- | :--- | :--- |
| **觸發方式** | 使用者提問 | 使用者下達任務 | **系統定時 Tick (心跳)** |
| **運作模式** | 單次對話 | 任務級迴圈 | **常駐背景工作** |
| **代表系統** | ChatGPT | Claude Code (現狀) | **KAIROS (未來)** |

### 9.2 沉默的失敗與「生成/評估者分離」架構

論文提出了一個非常嚴峻的警告 (Sec 11.6)：**78% 的 AI 失敗是無聲的 (Invisible)**。AI 不會報錯，它只是「自信地寫出有 bug 的程式碼」。

**💡 解決方案：Harness 的三位一體**
未來的優質 Harness 架構不能只有「執行」，必須包含三個獨立的邏輯層，且最好由不同的模型擔任：
1.  **執行者 (Generator)**：負責思考與寫 Code。
2.  **評估者 (Evaluator)**：負責審查執行者的輸出，檢查是否符合 Coding Style 或安全規範。
3.  **驗證者 (Verifier/LSP)**：利用確定性的工具（如編譯器、LSP、單元測試）來提供最後的真理。

> **學習點**：如果你在設計系統，不要只優化 Prompt。你應該設計一個 **「生成者與評估者對抗」** 的工作流，讓失敗在進入使用者視線前就被攔截。

### 9.3 終極挑戰：監督悖論 (Paradox of Supervision)

這是我們整份筆記中最具哲學深度的部分 (Sec 11.4)。論文指出，Agent 越強大，對人類長期的負面影響可能越大：

*   **現象**：短期內生產力爆發，但長期專案複雜度飆升。
*   **根源**：
    1.  **認知卸載 (Cognitive Offloading)**：人類因為信任 AI，不再檢查細節，導致對系統失去掌控。
    2.  **上下文碎片化**：因為 5 層壓縮管線，AI 看到的永遠只是專案的碎片，導致它不斷寫出重複、衝突的邏輯。
*   **未來的設計使命**：
    未來的 Agent 架構師必須思考：**「如何設計一個『強迫人類學習』或『維持專案全域透明度』的架構？」** 而不僅僅是追求「自動化程度」。
