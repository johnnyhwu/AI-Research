# Cerebras 企業知識庫架構學習筆記

## 前言

**文章出處**：Isaac Tai, Daniel Kim, Mike Gao,《How We Built Our Knowledge Base》, Cerebras, 2026年7月15日發表。

這篇筆記整理自對 Cerebras 內部知識庫系統（Cerebras Knowledge）技術部落格的深度研讀與討論。該系統每天處理超過 15,000 次查詢，整合 Slack、Wiki、程式碼倉庫、Jira 等異質資料源，服務對象包含人類使用者、自動化流程與 AI agent。

這份筆記以「學習地圖」的六個階段為骨架，依序拆解該系統的核心設計：**統一資料抽象層 → 非結構化資料處理 → 混合檢索與 Reranking → 查詢 Planning 與 Agent 編排 → 組織與 Scoping**（階段 0 為心智模型建立，不計入技術階段）。每個階段都包含文章原始的技術內容、第一原理拆解，以及筆記作者身為企業 RAG 系統開發者，結合自身實戰經驗所做的對照、質疑與延伸討論。

**這份筆記的重點主軸有三條**：

1. **窄腰設計（narrow waist）貫穿全系統**——不只出現在資料寫入層，也重複出現在讀取融合層、多工具輸出整合層、乃至組織 scoping 層，是理解整篇文章最重要的單一心智模型。
2. **檢索與推理必須解耦**——這是筆記作者結合自身系統痛點所推導出的核心洞察，也是本文與作者原有架構最大的差異點所在。
3. **這篇文章的架構本質是單輪 pipeline，不是 agentic loop**——文章教的是「怎麼把一輪檢索做扎實」，不處理「需要迭代或序列依賴的多步推理」，這個邊界判斷是筆記作者在討論過程中自行推導出的重要洞察，文章本身並未觸及。

## 第一節：核心心智模型——窄腰設計與兩條相反的路徑

### 系統要解的根本問題

企業 RAG 的難點不在向量搜尋本身，而在三件事的交集：知識散落在異質系統中、知識持續變動、不同使用者需要的答案範圍不同。Cerebras 系統的使用者不只是人類，還包含 automation 與 agent，這個前提暗中決定了後續許多設計（例如為何要有 MCP 介面、為何工具要盡量無 LLM）。

### 窄腰設計（The Narrow Waist）

整個系統的形狀是一個沙漏：

```
   多樣的寫入來源（寬）
  Slack / Wiki / Code / Jira / 自訂DB
        ↓ 各自的 connector 做正規化
  ═══════════════════（窄腰）
   單一 embeddings table
  (document, embedding, metadata, source, timestamp)
        ↓ 統一介面查詢
   多樣的檢索與編排策略（寬）
```

窄腰的價值在於：**腰以上和腰以下可以各自獨立演化，互不干擾**。新增資料源只需寫一個 connector，讀取端完全不用改；新增檢索策略只需在讀取端加，寫入端完全不用動。文章原文印證：「every source, from Slack threads to netlists, lands in the same embeddings table, and anything in that table is immediately queryable through the same interface」——netlist（晶片線路表）和 Slack 閒聊能用同一張表處理，正是窄腰的價值。

**重要延伸發現**：窄腰設計在整篇文章中至少重複出現三次，這是理解全文架構一致性的關鍵：
- **第一次（寫入端）**：多異質資料源 → 統一 embeddings schema
- **第二次（讀取融合端）**：多路檢索排名 → RRF 融合成單一排序
- **第三次（讀取多工具端）**：多異質工具輸出 → executor 正規化成共同 evidence schema

### 兩條路徑：寫入統一，讀取分化

- **寫入路徑**：追求「統一」，消除差異。不管來源多異質，最後都收斂成同一種 row，來源的特殊性被封裝在 connector 內部。
- **讀取路徑**：追求「分化」，擁抱差異。同一個 query 可能需要不同策略打不同資料源，靠 planner 動態決定。

這個「寫入統一 vs 讀取分化」的張力是全文所有技術細節的母題。讀取端還有一個「分化又收斂」的節奏：planner 分化成多路 → fan-out 平行執行 → RRF 再收斂回單一排序。

### 基礎設施選擇：單一 Postgres 的複利效應

系統把 embedding、原始摘要、metadata、以及**同步狀態**全部放在同一個 Postgres（pgvector，3072 維，HNSW 索引）。這個選擇的複利效應在後續 CocoIndex（程式碼嵌入）處理中特別明顯：因為同步狀態與 embedding 儲存在同一資料庫，每次 commit 只需 re-embed 變動的 chunk，不必重算整個倉庫。文章原文：「This worked especially well for us because the synchronization state and embedding store live in the same database.」

## 第二節：統一資料抽象層

### 偽統一 vs 真統一

一個容易犯的錯誤是「形式統一但語意分裂」：如果 embeddings table 為每個來源多加專屬欄位（`slack_channel`、`code_repo`、`jira_ticket_id`...），下游查詢邏輯仍需針對來源類型做 if-else 判斷，這不是真正的窄腰。

Cerebras 的做法是真正收窄到底：只有五個必然存在的欄位——`document`、`embedding`、`metadata`（不透明 JSON）、`source`、`timestamps`。下游邏輯永遠不需要判斷來源類型才能處理一行資料；`source` 只在最終呈現階段才被用到，不影響檢索、排序、融合的核心邏輯。

**檢查點**：下游的檢索/排序邏輯裡，若出現「if source == 'X' then...」的分支，代表窄腰還沒收乾淨。

### Connector 的最小契約：三件事

每個資料源只需定義三件事，且**只有這三件事**：
1. **what**——資料怎麼變成 document + metadata（一個 transform 函式）
2. **how to connect**——怎麼從來源系統拉資料（一個 fetch 函式）
3. **how often**——多久同步一次（可以像 Slack 一樣每個 channel 各自設頻率）

**關鍵點**：這三件事裡沒有「怎麼檢索」。Connector 的責任在窄腰以下就結束，完全不需要知道下游會用什麼方式使用這份資料。這是「寫入與讀取解耦」在資料抽象層的具體體現。

### Plugin 模式的組織紅利

因為契約夠窄，新增資料源可以完全下放給不熟悉核心系統的團隊自己做：開一個 PR，附上一個小型 Python module，只要輸出符合共享 schema 的 row，其餘全自動接上。技術抽象做得好，會直接降低協作成本；如果新增資料源需要核心團隊介入，代表抽象層漏水了。

### Metadata 不只是展示用途，是檢索訊號的一部分

容易被忽略的一點：`metadata` 欄位同時是檢索排序訊號的載體，不只是事後展示用的附加資訊。例如 age decay 機制依賴 metadata 裡的 timestamp。因此 metadata 的設計必須在 connector 階段就有意識地為未來的排序訊號做準備（時間戳、來源類型、作者/團隊資訊等），否則後續要重新處理整個 corpus 的代價很高。

## 第三節：非結構化資料處理——Slack 三部曲

### 為什麼純向量搜尋不夠：三種失效模式

1. **資訊密度被向量空間抹平**：embedding 壓縮過程對「內容有多少資訊」不敏感，只捕捉語意方向。廢話和乾貨若語意方向接近查詢，分數可能相近。
2. **短訊息在 cosine 相似度上系統性佔便宜**：短訊息向量方向集中明確；長訊息因包含多個子主題，向量是多個語意方向的加權平均，方向被稀釋。這是 embedding 表示法的內在幾何特性，不是換模型能解決的。
3. **訊息意義依賴上下文**：單獨一句「that fixed it」脫離對話毫無意義，這解釋了為何處理單位選在 thread 層級。

### 訊號互補的設計心法：正交性

好的混合檢索設計，是**先精確診斷每種失效模式的根因，再找一個訊號的數學性質恰好不受那個根因影響**，而非隨意疊加常見組合。四種訊號與其對應的失效模式：

| 失效模式 | 對應訊號 | 免疫原理 |
|---|---|---|
| 語意方向掩蓋內容價值 | IDF | 直接算詞彙稀有度，與向量幾何無關 |
| 短訊息佔便宜 | 全文搜尋 | 精確 token 匹配，不受長短影響 |
| 上下文依賴 | （靠處理單位選 thread） | 保留完整對話脈絡 |
| 答案會過期 | Age decay | 直接讀 timestamp，與語意無關 |

兩種訊號要疊加才有意義的前提是：它們的失效模式必須「正交」——若兩種訊號對同一種輸入都會失效，疊加沒有解決任何問題。

### Distillation：不要直接 embed 原始文字

系統的原始 transcript 不會被直接 embed。做法是先用 LLM 讀完整條 thread，抽出四樣結構化資料——一句話問題、摘要、解決方案、系統與程式碼參照——正規化成一致格式後才 embed。原始文字則另外走 Postgres 全文索引（GIN），保持字面可搜尋但不進向量空間。

**第一原理**：embedding 品質高度依賴輸入文字的形式一致性。原始 thread 長度、參與人數、離題程度差異極大，直接 embed 會讓向量在「什麼代表這個文件」上不可比較。Distillation 用固定 schema 把異質資料收斂成一致格式，讓每個被 embed 的文件在「代表性」上可比較。文章原文用詞是「normalized」而非「summarized」，強調目的是讓向量在同一表示空間裡有一致語意，而非單純縮短文字。文章實測：「accuracy increased significantly when the thread was normalized into a consistent format」。

**通用判準**：若原始資料形式差異很大，該問的不是「要不要摘要」，而是「有沒有一個固定 schema，能把異質資料收斂成一致格式」。

### Bursting：蒸餾的代價與補救

Distillation 是有損壓縮，重要的旁枝訊息可能因不在主線問答脈絡而被摘要忽略。Bursting 的做法：定義 burst 為「同一作者的連續訊息」，單獨拿出來、前綴 thread 主題當 context 後單獨 embed。

**品質門檻**（每條各防一種噪音）：
- IDF ≥ 4.0——確保包含稀有詞，非廢話
- 長度 ≥ 200 字元——確保足夠內容支撐有意義的向量
- 有 reaction 加分——用社交訊號間接驗證價值

**工程心法**：與其事後靠 rerank 補救垃圾資料排太前面，不如寫入時用低成本規則先過濾一批。寫入時過濾是一次性成本，rerank 是每次查詢都要付的成本，越早擋噪音越省。

**具體範例——一條 thread 最終產生的 embedding 數量**：以一條包含 4 位發言者、5 則訊息的 thread 為例，最終會產生「1 筆 thread 級（distillation 產物） + N 筆通過品質門檻的 burst 級」embedding，全部寫入同一張表，`metadata` 標記類型為 `slack_thread` 或 `slack_burst`。未通過門檻的 burst（如純社交填充語）不會被 embed。

**與 Contextual Retrieval 的關聯**：Anthropic 的 Contextual Retrieval 核心主張是為每個 chunk 生成客製化上下文前綴，避免脫離語境的孤立片段。Bursting 是同一原理的輕量版：不為每個 burst 各跑一次 LLM 生成上下文，而是直接把 thread 主題貼上去當前綴，用更低成本達到類似效果。

## 第四節：混合檢索與 Reranking 的數學基礎

### 為何不能直接比較不同檢索器的分數

不同檢索器的分數尺度、單位、分布形狀都不同（BM25 分數範圍不固定，cosine 相似度在 -1 到 1 之間），無法直接相加比較。常見的「正規化到 0-1 再加權相加」也有陷阱：正規化依賴最大最小值的定義方式，容易讓同一份資料在不同查詢批次中的相對權重飄移。

### RRF（Reciprocal Rank Fusion）公式與運作原理

```
RRFscore(d) = Σ  weight_i / (k + rank_i(d))
              i∈檢索器
```

其中 k=60（平滑常數，源自 Cormack et al. 2009 論文建議值），weight 預設 1.0。若某檢索器未撈到該文件，該項為 0（不貢獻也不懲罰）。

**核心洞察**：RRF 只用「名次」不用「分數」，完全繞開了分數尺度不可比的難題——名次是所有檢索器天然具備且天然可比較的整數。

**k 值是設計哲學的旋鈕**：k 越大，頭部名次之間的分數差距被壓得越平，系統越傾向「多路都上榜」累積分數贏過「單路衝第一」，即偏好共識勝過單一強訊號；k 越小，第一名的獎勵越懸殊，系統越傾向相信單一檢索器的強烈信號。k=60 代表 Cerebras 選擇「寧可相信多個獨立訊號的共識，也不要被單一路的巧合命中牽著走」。

**動筆驗算範例**：三份候選文件在兩路排名不同時，RRF 分數計算展示「兩邊都排第三的平庸穩定文件」可能贏過「一邊第一、一邊第八的強訊號文件」，這正是 RRF「獎勵共識、懲罰單路強訊號」的具體數學效果。

### IDF 與 BM25 的關係澄清

**核心區分**：IDF 是 BM25 的一個組成零件，而 Cerebras 文章把這個零件單獨抽出來當成一個獨立訊號使用，兩者角色不同：

| | BM25 | 文章的 IDF 用法 |
|---|---|---|
| 角色 | 完整的 sparse 排序函數（IDF × TF飽和項 × 長度正規化項） | 單一加權訊號 / 品質閘門 |
| 輸出 | 一份排名 | 一個乘數，調整其他訊號 |
| 用途 | 完整的 sparse retrieval 那一路 | 額外壓低「語意上靠近但全是廢話詞」的結果 |

文章中的「full-text search」對應到熟悉的 sparse retrieval（BM25）那一路；IDF 是額外抽出來、跨所有路使用的品質訊號。

**IDF 完整公式（BM25 版本）**：
```
IDF(term) = ln( (N - n(term) + 0.5) / (n(term) + 0.5) + 1 )
```
N 為語料庫總文件數，n(term) 為包含該詞的文件數。詞越稀有（n(term) 越小），IDF 越大。

**BM25 完整公式**：
```
BM25(D,Q) = Σ  IDF(term) × [ f(term,D)×(k1+1) ] / [ f(term,D) + k1×(1-b+b×|D|/avgdl) ]
          term∈Q
```
k1、b 為可調參數（常見預設 k1=1.5, b=0.75），分子隨詞頻增加但飽和，分母對長文件做懲罰。

> **【個人實戰洞察】IDF 獨立閘門的應用價值**
> 若系統已經在做 BM25 + 向量的混合檢索，IDF 其實已經隱含在 BM25 內部。但可以把 IDF 額外再抽出來，當成一個獨立的過濾/加權訊號，專門用來壓低「語意上靠近但全是廢話詞」的結果。這在處理對話型資料（大量社交填充詞）時特別有用：BM25 排名本身不會主動幫忙過濾掉那些在 dense retrieval 裡浮上來的廢話內容，但一個獨立的 IDF 閘門可以額外做這件事，兩者不衝突、可疊加使用。

### 融合後的三個收尾動作

**去重與限額（per-file cap）**：同一份文件可能被切成多個 chunk，若多個 chunk 都排進候選集合前段，會佔滿名額、降低資訊來源多樣性。做法是限制同一份文件最多貢獻 N 筆進入候選集合，確保候選集合涵蓋多樣化的資訊來源，而非一份文件的多種切法。

**Reranker 打分**：RRF 排序是純統計融合，不理解 query 的實際語意；reranker（cross-encoder）把 query 與 document 逐一配對，直接輸出相關性分數，能做到 RRF 做不到的語意層面判斷。

> **【技術澄清】Reranker 呼叫次數的精確理解**
> 「一次呼叫」reranker 指的是打包成一次批次（batch）API 呼叫，內部仍對每個候選文件各自進行一次 `(query, document)` 配對推論（例如 20 個候選即 20 次配對），並非把所有候選塞進同一個 context 一次性處理。這與 synthesis 階段的 LLM 呼叫本質不同：
>
> | | Reranker | Synthesis LLM |
> |---|---|---|
> | 輸入格式 | 每次只看 (query, 單一document) | 一次看 (query, 所有候選document) 全塞進同一context |
> | 模型任務 | 打分數（分類/回歸） | 生成連貫文字（生成任務） |
> | 計算量級 | 小模型，快速 | 大模型，context長，較慢 |
>
> 處理線是「漸進昂貴」：越後面的步驟單次呼叫成本越高，因此用前面便宜的步驟（RRF 融合、去重限額）盡量縮小候選集合，把最貴的運算（synthesis）留給少量高信心候選。

**補回上下文**：拿到最終 top 10 後，若贏家是文件的某個 section，額外撈回相鄰的兩個 section 一起呈現，避免 chunking 把前提條件、注意事項切到隔壁 chunk 而遺失。此步驟只發生在 rerank 之後，只對最終要呈現的少量結果做，避免在候選階段浪費資源。

> **【個人實戰問題與解法】RRF 的結構性弱點——投影片檢索場景**
>
> **問題描述**：在處理投影片資料時（每頁投影片透過 VLM 產生 summary 再 embed，搭配 hybrid retrieval），常出現 ground truth 頁面在 dense retrieval 排第一名，但 sparse retrieval 完全沒撈到該頁面（反之亦然）。這導致 RRF 融合後正確答案排名不佳。暫時的權宜解法是「永遠保留 dense 和 sparse 各自的第一名」。
>
> **根因分析**：RRF 的「獎勵共識」機制隱含一個假設——每一路都至少「看得到」正確答案，只是排名高低不同。當正確答案在某一路完全缺席（連候選都沒進）時，RRF 對那一路的貢獻是 0，「共識」機制反而變成「懲罰」，正確答案只拿到單路分數，可能輸給兩路都排名普通但都有出現的雜訊文件。
>
> 投影片場景特別容易觸發此問題，因為 dense 和 sparse 兩路**共用同一個不穩定的中介來源**（VLM summary）。VLM 生成 summary 時用詞不穩定（同一張圖可能寫「revenue growth」也可能寫「sales increase」），導致 sparse 路對字面詞彙敏感、容易因用詞不巧而完全掛零；而兩路訊號源頭一致，破壞了「訊號要正交」的前提。
>
> **建議解法（由淺到深，可組合使用）**：
> 1. **提高各路召回深度**：許多「單路缺席」實為候選截斷所致（正確答案排在截斷線之外），將各路 top-K 拉深（如 top 50-100）可救回被截斷的正確答案，且 RRF 對深處名次貢獻小，不易引入噪音。
> 2. **修正表示層，讓兩路訊號源頭不同**：sparse 路不應只 index VLM summary，應額外加入投影片頁面的原始文字（OCR 抽出的標題、bullet、圖表標籤），原始文字用詞穩定，不受 VLM 選詞不穩定影響；dense 路維持吃語意化的 VLM summary。讓兩路的失效模式真正正交。
> 3. **用 reranker 補 RRF 結構弱點**：各路取深召回後取聯集（而非只信 RRF 分數），交給 cross-encoder reranker 重新逐一打分。Reranker 對 query-document 直接配對評估，不管來源路數，只要正確答案進了聯集就有機會被排到最前面，天生免疫「單路缺席」問題。此為「永遠保留兩邊第一名」這個權宜解法的一般化版本。

## 第五節：查詢 Planning 與 Agent 編排

### Planner 存在的意義：成本與延遲的按需分配

若無 planner、每次查詢一律呼叫所有工具，會導致：不需要的工具也被觸發（延遲、成本浪費），以及不相關工具的雜訊結果可能拉低整體排序品質。Planner 的第一原理是把「該用哪些工具」從固定規則變成動態決策，讓成本與延遲跟查詢的實際需求成正比。

### Planner 的輸入輸出結構

三項輸入：
1. **工具目錄（compact description）**——只需精簡的能力說明（這個工具擅長回答什麼類型問題），不需要完整技術文件或內部實作細節
2. **使用者的 query**
3. **active scope**——當前鎖定的查詢範圍（對應第六節 Projects 概念）

輸出：一組工具選擇清單，交給 executor 平行 fan-out。

### Planner 要故意做得比它服務的工具更便宜

Planner 執行的是分類任務（輸出離散選項），不是生成任務，因此可用小模型、結構化輸出，推理成本與延遲低。這是整條處理線裡最便宜的一環。**架構原則**：決策層要故意比執行層便宜，否則「用來省錢的決策」本身若太貴，設計就自相矛盾。

### 工具對 Planner 而言永遠是黑盒子

Planner 只需要知道工具的「能力描述」，不需要知道工具內部是簡單查表（lookup）還是需要額外推理（compute）。這使得不同性質的工具可以在同一套 planner 邏輯下被平等看待。

> **【個人實戰問題與解法】SQL 場景的 Lookup vs Compute 區分**
>
> **問題描述**：在處理需要 SQL 查詢的問題時，流程通常是「檢索表格 schema → 一次 LLM 做 SQL reasoning 取出相關數據 → 再一次 LLM 基於數據產生文字回答」。這與文章「單次 synthesis 產生最終答案」的做法看似矛盾——若檢索工具需要內部推理才能取得證據，是否代表文章的單輪 synthesis 架構不適用？
>
> **核心區分——Lookup vs Compute**：
> 文章列出的所有工具（`search`、`search_slack`、`search_code`、`who_knows` 等）檢索的對象都是「已經被 embed 或 index 好、預先算好、躺在那裡等人拿」的靜態資料，檢索動作本質是唯讀撈取，撈出即為最終形態的證據，因此可以直接進入融合排序而不需額外推理。
>
> SQL 場景本質不同：「把相關數據取出來」這件事本身需要當場生成 SQL、執行、才能得到答案，這是一個不可省略的推理步驟（compute），而非唯讀撈取（lookup）。
>
> **解法——判準與重構方向**：
> 一次 LLM 呼叫是否該保留、以及該放在管線的哪個位置，取決於它的產出性質：
> - 若產出是**證據**（下游還要再加工）→ 可以存在，但應**藏在檢索工具內部**（例如 SQL 生成該留在 `search_sql` 工具內，作為取得查詢結果數據的必要步驟）
> - 若產出是**面向使用者問題的部分答案**（如針對單一資料源提前生成的摘要文字）→ 應該**砍掉，延後到管線最末端的單次 synthesis**
>
> 因此正確做法不是「SQL reasoning 該不該存在」的二選一，而是保留 SQL 生成這一步（藏進工具內部，對外只暴露它吐出查詢到的數據列），但拿掉「基於這批數據單獨產生文字回答」這一步，改為把數據當作證據交給最末端的單次 synthesis，與其他來源（如向量檢索到的文字片段）一起做跨源推理。
>
> Planner 眼中，`search_sql` 與其他工具一樣只是一個依「能力描述」被選用的黑盒子，不需要知道其內部是 lookup 還是 compute。若使用者問題同時需要查數字與查政策說明，planner 應同時選出對應的兩個工具一起 fan-out，兩邊各自吐出證據，在最終 synthesis 時才一起被交叉推理。

### MCP 與 Web UI：同一套原語，兩種介面設計

**MCP 介面**刻意不包含 planner，只暴露檢索原語本身，工具「intentionally simple and as LLM-free as possible」，由外部 agent（如 Claude Code）自行決定呼叫順序與結果組合。

**Web UI 介面**則把 planner、executor、synthesizer 全部包起來，對使用者呈現為單純的問答介面。

**設計原因**：MCP 的使用者是另一個具備推理能力的 agent，不該替它預先做決策——若工具背後偷藏一層 planner，會讓外部 agent 對工具行為的預期失控（工具行為不再是輸入的確定性函數，而取決於一個外部 agent 看不到的隱藏決策層）。Web UI 的使用者是人類，人類不會自己組合工具呼叫，所以系統必須替他包好完整流程。

**延伸啟發**：若系統未來想被其他 agent 呼叫使用，必須把「檢索原語」和「編排決策」拆成兩層可獨立暴露的介面，這是「檢索與推理解耦」原則在更高一層（編排層）的重演。

### Executor 的兩個職責

- **平行執行**：選中的工具同時打出去。能平行化的前提是工具之間沒有依賴關係（工具本身是無狀態的檢索原語）。
- **正規化成共同 evidence schema**：不同工具吐出的原始格式差異很大，executor 需要統一成 synthesis 能理解的共同格式（含 scores、recency、source hints），這是窄腰設計在讀取端多工具輸出整合層的第三次出現。

## 第六節：組織與 Scoping（Projects）

### 為何「search everything」在規模變大後失效

Corpus 越大，噪音對排序品質的侵蝕越嚴重——原本不相關的文件只要跟 query 沾到一點語意邊或字面邊，就有機會擠進候選集合前段，把真正該出現的答案往下推。這不是檢索演算法變差，而是「全域搜尋」這個假設本身隨 corpus 成長會系統性惡化訊噪比，且此現象具隱性——小規模測試正常，上線後隨資料成長悄悄劣化，難以定位是哪個技術環節出問題。

### Projects：分組而非複製

一個 project 是一組具名的資料源 bundle。關鍵字是「引用（reference）」而非「複製（duplicate）」——同一個資料源可被多個 project 引用，不需要複製資料。這代表 project 本質上是查詢時的邏輯過濾條件（例如對 metadata 中的 source 欄位做範圍限制），而非新的資料容器。此設計避免了資料更新的一致性風險與儲存成本隨 project 數量線性成長的問題，也再次體現「寫入端與讀取端獨立演化」的原則——新增一個 project 不需碰觸任何資料寫入邏輯。

### Scoping 與其他階段的交互關係

- **與 Reranking 的關係**：Scope 過濾應在 RRF 融合之前就先執行，而非融合、rerank 完才篩選，否則會浪費排序運算在 scope 外的資料上，且可能讓 top-K 名額被 scope 外文件佔用。
- **與 Planner 的關係**：Active scope 是 planner 決策的輸入之一，不只是後續檢索的過濾條件——它同時限縮「查哪裡」與「該不該用某個工具」（例如某 project 沒有掛任何自訂資料庫，`search_sql` 這類工具大概率不該被考慮）。

### Onboarding 綁定預設 Project：解決冷啟動問題

若要求使用者自行選擇 scope，等於把「系統該懂的組織知識」轉嫁給使用者，這是本末倒置的設計。做法是把預設 project 綁在使用者 profile 上，新使用者無需先理解公司內部有哪些資料源分組，即可自動獲得高訊號答案。這是「把系統複雜度留在系統裡，不外洩給使用者」原則在使用者體驗層的體現。

## 第七節：系統架構的邊界——單輪 Pipeline vs Agentic Loop

這一節的內容並非源自文章本身，而是在討論過程中，對照個人開發經驗所發現、文章完全未觸及的架構邊界，其重要性足以獨立成節記錄。

### 文章架構的本質：單輪、無迴圈

文章描述的完整流程是：

```
query → planner決策(一次) → 平行執行(一次) → synthesis(一次) → 結束
```

這是單輪（single-round）架構，結構上不支援兩種常見的複雜需求：

1. **「發現資訊不夠再迭代」需要迴圈**：synthesis 或某個中間步驟須能判斷「證據不夠」，然後回頭再發一次檢索。文章的 executor 是 fan-out 完就結束，沒有「evaluate → 不夠 → 再來一輪」的回饋路徑。
2. **「先查某個再查某個」需要序列依賴**：文章明確採用平行執行，平行的前提正是工具彼此獨立、互不依賴。若工具 B 的輸入依賴工具 A 的輸出，在「平行 fan-out」模型下無法表達。

### 兩個問題的共同根因

「迭代」與「序列依賴」表面是兩件事，本質上是同一件事——**系統缺少「狀態」和「多步推理」**。單輪架構的隱含假設是「這次 query 需要哪些證據，在一開始（planner 那一刻）就能被完全決定」，但真實情況是「我還需要查什麼」這件事，往往只有看過第一輪結果之後才會知道，這是本質上的認知順序問題。

### 兩種範式的精確區分

| | 文章的架構（RAG pipeline） | 需要迭代/序列時的架構（Agentic loop） |
|---|---|---|
| 決策次數 | 一次（planner決定完不再變） | 多次（每輪結果可能觸發下一輪決策） |
| 工具間關係 | 獨立、可平行 | 可能有依賴、需要序列 |
| 何時停止 | 固定流程跑完就停 | 由判斷邏輯動態決定「夠了嗎」 |
| 本質 | 一個流程（pipeline） | 一個會自己決定下一步的 agent |

文章做的嚴格而言不是「agentic RAG」，而是「用 LLM 做動態路由的傳統 RAG pipeline」。Planner 雖用了 LLM，但智慧僅止於「這次該用哪些工具」，不包含「這次夠不夠、要不要再來一輪」。「用了 LLM 做決策」不等於「agentic」，agentic 的核心特徵是迴圈與狀態，而非是否有 LLM 參與決策。

### 文章對迭代/序列問題的殘餘價值

文章給的不是「怎麼做迭代」的答案，而是「迭代迴圈裡每一輪該怎麼做好」的答案。若要建構支援迭代與序列依賴的系統，形狀大致為：

```
while 資訊不夠 or 還有下一步要查:
    agent決定下一步要查什麼、用什麼工具    ← 文章未觸及
    執行檢索                              ← 文章階段1-3教如何做好
    評估「這次查到的夠不夠、下一步要查什麼」  ← 文章未觸及
    合併進累積的證據池
synthesis 讀完整個累積證據池產出答案       ← 文章有講（單輪版本）
```

外層 while 迴圈與「決定下一步」的推理，屬於 agentic 系統設計範疇，文章完全沒有觸及。但迴圈內部「怎麼把單輪檢索做好」的技術（資料處理、多路訊號融合排序、單輪內的 planner 工具選擇），完全適用且不過期——**單輪架構的品質，決定了迴圈裡每一次迭代的品質；迴圈的存在與否，決定了系統能不能處理需要多步推理的複雜問題**。這兩層是正交的關係。

### 判斷框架：如何決定該往哪個方向補強

**判斷一：「資訊不夠」的判斷發生在哪一層？**
- 若是 synthesis 讀完證據後才發現不夠 → 需在 synthesis 之後加入「self-critique」或「sufficiency check」步驟，讓其輸出「還缺什麼」，再回頭餵給 planner 觸發下一輪。
- 若是使用者追問才觸發 → 屬於對話管理範疇，非需要迴圈的迭代問題，單輪 pipeline 每次追問重跑一次即可。

**判斷二：序列依賴是「有限幾種固定模式」還是「真的需要 agent 自主規劃」？**
- 若模式有限（如固定「先查人 → 再查該人的PR」兩步流程）→ 不需要完整 agentic loop，可將此序列寫成一個複合工具，內部固定執行順序，對外仍暴露為 planner 眼中的單一黑盒工具（沿用「工具對 planner 是黑盒子」原則）。
- 若序列依賴是開放式、無固定模式、需要 agent 臨場判斷下一步 → 才是真正需要 agentic loop 的情況，此時文章整套 planner/executor/synthesis 架構最多只能降級為 agent loop 中「單輪執行」的子模組，外部須另包一層真正決定是否繼續、下一步查什麼的推理迴圈。

## 第八節：個人系統對照回顧——從舊架構到新理解的轉變

這一節統整討論過程中，個人原有系統設計與文章架構之間的落差，以及由此推導出的關鍵重構方向。

### 原有架構的問題診斷

原有系統以「領域/客戶」作為分化維度，每個領域綁定固定的多種 retrieval 方法（例如某領域固定用三種檢索方法打三種資料庫再合併）。這種設計在規模擴大後暴露兩個問題：

1. **不同領域難以共用 retrieval 方法**：即使不同領域擁有相同類型的資料或問題類型，因為 retrieval 方法綁定在領域維度上，無法跨領域共用。
2. **同領域內新增一種資料類型，就需開發一種新檢索方法**：分化粒度綁死在領域層級，無法對單一 query 的意圖做動態调整。

**根因判定**：這兩個問題的共同根因是「檢索與推理耦合」——原有的融合流程是「三種檢索 → 各自呼叫對應推理方法（如SQL推理、圖文推理）→ 各自產生摘要 → 最後用一次LLM合併三份摘要」，等於每一路檢索都內建了完整的推理與摘要流程，檢索與推理在架構上無法分離，導致無法跨領域共用（推理邏輯焊死在檢索上），也無法用動態 planner 取代領域綁定（因為每個「路」不是純粹的檢索原語，而是完整的問答套件）。

### 文章架構帶來的關鍵重構方向

**核心轉變**：把檢索與推理解耦——檢索工具只負責吐出原始證據（不做推理、不做摘要），推理只在管線最末端的單次 synthesis 發生。

```
原有架構：
  檢索A → 推理A(如SQL) → 摘要A ┐
  檢索B → 推理B(如圖文) → 摘要B ┼─→ LLM合併 → 最終答案
  檢索C → 推理C        → 摘要C ┘

文章架構：
  檢索A(唯讀撈取) ┐
  檢索B(唯讀撈取) ┼→ 融合排序(RRF, 純數學) → rerank(1次小模型) → 單次synthesis → 答案
  檢索C(唯讀撈取) ┘
```

**此轉變帶來的具體效益**：
- **解決問題1（跨領域難共用）**：解耦後，檢索工具變成乾淨、無狀態、不綁推理的原語，天生可跨領域共用。
- **解決問題2（新增資料類型要開發新檢索方法）**：搭配 planner 機制，新增資料類型時只需新增「一個檢索工具」並更新其能力描述，planner 自動學會在合適的 query 選用它，不需修改既有推理邏輯。
- **修復跨源推理能力**：原架構中「每路各自完整推理再摘要」的做法，會導致沒有任何一個 LLM 同時看到全部證據，若正確答案需要跨源交叉推理（如「A的一句話 + B的一張圖」），原架構結構性地做不到；文章架構的單次 synthesis 讓所有證據攤在同一 context 裡一次推理，天生支援跨源推理。
- **減少誤差累積**：原架構「摘要的摘要」（每路摘要 + 最終合併摘要）是兩層有損壓縮的疊加，下游無法察覺上游丟失了什麼；文章架構只在最末端壓縮一次，證據在此之前保持原始形態。

**例外情況的保留**：並非所有「檢索前的推理」都該被消除。若某類檢索的本質是「計算（compute）」而非「查表（lookup）」（如 SQL 生成需要當場推理才能取得數據），該次推理應保留，但需藏進檢索工具內部，只對外暴露「吐出查詢到的證據」，而非「產生面向使用者問題的部分答案」（詳見第五節 Lookup vs Compute 判準）。

---

## 結論

這篇文章的核心貢獻，是完整示範了一套「窄腰設計」如何在異質企業資料場景中，同時解決寫入端的可擴展性與讀取端的檢索品質問題——這個設計原則從資料抽象層、到讀取融合層、到多工具編排層、再到組織 scoping 層，反覆出現了四次，是全篇最值得內化的單一心智模型。

透過與個人現有企業 RAG 系統的對照討論，最重要的收穫並非文章本身羅列的技術細節（如 RRF 公式、distillation 流程、bursting 機制），而是這些技術細節共同指向的一個更根本的架構原則：**檢索與推理必須解耦**。原有系統以領域為分化維度、每路檢索綁定完整推理與摘要流程的設計，看似合理，實則同時造成了「跨領域無法共用」與「新資料類型需重複開發」兩個問題，而這兩個問題的共同解法，就是把推理徹底移出檢索路徑、只在管線末端的單次 synthesis 統一發生。

同樣重要的是透過討論所釐清的**架構邊界**：文章描述的整套 planner-executor-synthesis 架構，本質上是一個單輪（single-round）pipeline，不具備處理「證據不足需要迭代」或「查詢之間存在序列依賴」這類複雜推理需求的能力，因為其平行 fan-out 的設計隱含了工具彼此獨立、且所需證據在查詢當下即可被完全決定的假設。這意味著文章教授的所有技術（混合檢索、reranking、資料蒸餾、動態工具選擇），本質上是在回答「如何把單輪檢索做扎實」，而非「如何做多步 agentic 推理」——後者需要另一層具備狀態管理與動態規劃能力的迴圈架構，文章完全沒有觸及。這個邊界判斷的價值在於：它清楚劃分了「文章技術直接適用的範圍」與「需要另外尋求 agentic RAG 相關知識來補強的範圍」，避免將單輪 pipeline 的優化誤認為能解決本質上需要多步推理的問題。

最後，這次討論也驗證了一個閱讀技術文章的方法論：**技術細節的價值往往不在於直接照搬，而在於用自己的實戰場景去逐一檢驗其背後的第一原理是否成立、以及在哪些邊界條件下會失效**（如投影片檢索中 RRF 因訊號源頭不正交而失效的案例）。這種對照式的閱讀，比單純理解文章本身能帶來更深刻且可直接落地的架構判斷力。
